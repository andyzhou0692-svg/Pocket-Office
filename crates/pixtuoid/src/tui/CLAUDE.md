# pixtuoid/tui — terminal renderer agent guide

The **terminal painter**: ratatui `App` + `TuiRenderer` (its inherent `render`
flush — the core `Renderer` trait was retired #483). Owns the half-block flush, the widgets (footer/wall-display/theme-picker
UI/tooltip/help/dashboard/Sources panel), mouse hit-testing, the version popup,
the crossterm event loop + terminal lifecycle, and the per-session UI models
(dashboard, Sources/connection). This is where the `pixtuoid-scene` engine's pixel
buffer becomes terminal cells.

**The render + simulation ENGINE is its OWN crate
[`pixtuoid-scene`](../../../pixtuoid-scene/CLAUDE.md)** (the DAG is
`pixtuoid-core ← pixtuoid-scene ← pixtuoid`) — layout, pose/motion/pathfinding,
the pixel pass (`render_to_rgb_buffer`), the color theme MODEL, pets, chitchat,
frame_cache, embedded_pack. `tui` and `floating` are sibling thin painters over
the `pixtuoid-scene` crate; neither depends on the other. When a painter entry
below references engine internals (`pixtuoid_scene::pixel_painter`,
`pixtuoid_scene::theme`, `pixtuoid_scene::motion`, …) follow the link to the
[scene guide](../../../pixtuoid-scene/CLAUDE.md) for the deep "why."

Parent guides: binary [`../../CLAUDE.md`](../../CLAUDE.md); workspace
[`../../../../CLAUDE.md`](../../../../CLAUDE.md); headless lib
[`../../../pixtuoid-core/CLAUDE.md`](../../../pixtuoid-core/CLAUDE.md); the engine
[`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md).

## Known sharp edges (don't be surprised by these)

- **`draw_scene` is called through `TuiRenderer`** (its inherent `render` flush; the core `Renderer` trait was retired #483), which owns the cross-frame state (per-floor `FloorCtx` {FrameCache/Router/OccupancyOverlay/PoseHistory/motion/light} + the `RgbBuffer`) and assembles a per-frame `DrawCtx` borrow that carries that `FloorCtx` as ONE `store` field (was the flat FrameCache/Router/OccupancyOverlay/PoseHistory list) + a separate `buf` (disjoint sibling), plus theme + mouse state. It delegates the world render to `pixtuoid_scene::pixel_painter::render_to_rgb_buffer` (the shared engine seam — see [`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md)), then does the terminal-only half-block flush + widgets + footer. `draw_scene` returns `Result<Option<Layout>>` — the computed layout is cached on `TuiRenderer.cached_layout` so hit-test functions can use it without recomputing. During floor transitions, `cached_layout` is cleared to `None`.
- **The version popup's URL click-rect (`version_popup_url_rect`) derives its offsets from the SAME `PANEL_PAD_*` consts the painter insets by** (`url_x = popup_x + PANEL_PAD_X + URL_PREFIX.len()`, edges clipped at `±PANEL_PAD_*`) — keep them in lockstep (the phantom-browser-launch regression class).
- **The board's ★ Star click target is `hud::star_hit_rect`, the same phantom-launch class** — it derives the precise `★ Star` span from the SAME board geometry the L1 painter uses (`cell_x = scene.x+2`, `cell_y = scene.y+1`, right-flushed to `BOARD_W`, star text = `BOARD_STAR`) and clips to the scene (`None` on a too-narrow terminal). The caller (`tui/mod.rs`) still gates on `cached_layout().is_some()`. It REPLACED the loose `hit_test_branding` (fired on all of cols `1..31`). Keep the paint + the hit-rect deriving from the same `BOARD_W`/`BOARD_STAR` — the `debug_assert` in `paint_wall_display` pins the brand-fits invariant the pairing needs.
- **Every crossterm key event passes `should_dispatch_key` first** (`tui/mod.rs`): Windows delivers Press AND Release (and Repeat) per keystroke, so only `KeyEventKind::Press` dispatches — otherwise toggle keys double-fire there (`p` pauses then instantly unpauses). Unix only emits Press, so the gate is a no-op locally.
- **The `w` walkable/approach/route debug overlay is dev-only** — its dispatch arm is `#[cfg(debug_assertions)]`-gated, so release builds silently ignore `w`. Don't "fix" a report that `w` does nothing in an installed binary.

## Where to look

- "How is the office rendered (pixel pass → terminal)?" → the world render is `pixtuoid_scene::pixel_painter::render_to_rgb_buffer` (see [`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md) for the pixel-pass split). The terminal-flush pass (`half-block + widgets + status footer`) is **`tui::renderer::draw_scene`**. `draw_scene` is the ONE consumer still on raw `render_to_rgb_buffer` (it needs the full `PixelPassResult` + immutable coffee borrows mid-flush); every headless surface goes through the shared `pixtuoid_scene::floor::render_floor` seam (#423) — the floor-slide calls it directly (projected scenes, caller-side eviction), while `floating::offscreen::OfficeRenderer` and the web hero each own a `pixtuoid_scene::floor::FloorSession` (the scene-owned session whose `render()` also runs the dual eviction); `TuiRenderer` composes the same session halves as `Vec<PerFloor>` + one `PerOffice`. `examples/snapshot/` saves the ratatui `TestBackend` via `save_backend_as_png` (a ▀-compressed PNG with footer).
- "How does the neon wall board work?" → the pixel-buffer half (dark panel + pulsing cyan border) is `pixtuoid_scene::pixel_painter::background::lighting.rs::paint_neon_panel`; the TEXT is a backend-agnostic MODEL, `pixtuoid_scene::board::build_board(counts, uptime, floor, gateway) -> BoardModel` (see the scene guide), which `widgets/hud.rs::paint_wall_display` renders as 3 rows of ratatui text by mapping each segment's `BoardTone` → color via `board_tone_color`: **L1** `model.brand` + the `★ Star` CTA (`model.star`, right-flushed; its precise click target is `hud::star_hit_rect`, opening `REPO_URL`); **L2** the plain-English **mood pulse** (`model.mood` — one `{glyph}{count} {word}` per non-zero present state, the ▲ "needs-you" beacon LEADING, NUMERIC + abbreviated to fit, from the SAME `scene_stats` the footer reads); **L3** office context (`model.context`: uptime `Dim` + floor breadcrumb + the `⬢gw` gateway chip). The floating window + wasm site paint the SAME `build_board` model their own way (AA / DOM), so all three surfaces can't drift. The board's cell-origin + WIDTH pin to the painted panel's dark INTERIOR — `hud::BOARD_W = pixtuoid_scene::pixel_painter::NEON_PANEL_INNER_W` (the outer `NEON_PANEL_W` minus its `NEON_PANEL_BORDER` on each side) and `hud::board_cell_origin` starts text at `NEON_PANEL_INNER_X` — so the lit text sits INSIDE the glowing frame (laying it to the full outer width overran it — the board-overflow fix; `paint_neon_panel` reads the SAME border const, so the dark cells it leaves == the cells text may fill). `board_cell_origin` is the ONE origin `paint_wall_display` AND `star_hit_rect` share. **The scrolling ticker was removed** (0.12.0) — `TickerQueue` + the `neon_ticker` theme color are gone; `marquee_window`/`MARQUEE_*` survive (dashboard/connection selected-row scroll).
- "How does the theme PICKER UI work?" → the public Pocket Office is fixed to `200West`, so `[t]` is unbound and the picker is not exposed. The dormant theme MODEL (`Theme`, `ALL_THEMES`, `SourceColors`/`ApplianceColors`) and picker implementation remain for a future product decision. `config::resolve_theme` is the product lock and always returns `GOLDMAN`, whose canonical display name is `200West`.
- "How are popups framed?" → **every popup is borderless** (no outline) and shares ONE backing: `widgets/mod.rs::paint_card_backing(f, area, theme)` = a flat single-color drop **shadow** (a hand-rolled fg+bg dim — NOT ratatui's `Block::shadow`, which smears over half-blocks) + `Clear` + a solid `tooltip_bg` fill. **The shadow is a deliberately FLAT offset silhouette, not a soft penumbra** (`cast_drop_shadow`): the card's own rect is darkened by ONE uniform `SHADOW_FACTOR` (0.42) and offset `SHADOW_OFFSET` (1) cell down-right; the card paints over its own cells afterward, leaving an L-band (right column offset down, bottom row offset right, meeting at the corner) — all ONE color. The **bottom-most row is rendered TOP-HALF only** (`dim_cell`'s `top_half_only`: dims `fg`, leaves `bg` lit) so the bottom band reads as a 1px contact line instead of a full 2px cell, while the vertical right strip stays full cells. This is a pinned user preference (matches a supplied reference — flat single color, offset, darker, thin bottom); do NOT "restore" a gradient/falloff or per-band factor, and do NOT make the bottom a full 2px cell — the offset silhouette with the 1px top-half bottom is the intended look (`panel::borderless_panel_casts_a_flat_offset_shadow` pins it). Both card kinds delegate to it, so the look (incl. the shadow) has a single definition and can't drift: `tui/widgets/panel.rs::borderless_panel` calls it, then insets by a uniform `PANEL_PAD_X`/`PANEL_PAD_Y` (the breathing room that replaces the border — re-exported from `widgets`), draws an optional bold brand-colored title row, and returns the inner content `Rect` (Help / theme-picker / version-popup / dashboard / connection route through it, each `centered_in` adding `2×PANEL_PAD_*`); the hover/info tooltips (`tooltip.rs::framed_tooltip`, a `Padding::uniform(1)` text-only Block — the bg/clear/shadow now come from the backing) call it too. Chitchat speech bubbles are deliberately NOT this primitive (a bare wrapping `Paragraph`, no `Clear`/block) so they get no shadow; long theme-pack dialogue wraps within the scene instead of clipping. The title moved from the old border into the body. **The version popup's URL click-rect (`version_popup_url_rect`) derives its offsets from the SAME `PANEL_PAD_*` consts the painter insets by** (`url_x = popup_x + PANEL_PAD_X + URL_PREFIX.len()`, edges clipped at `±PANEL_PAD_*`) — keep them in lockstep (the phantom-browser-launch regression class).
- "How does the version popup render?" → `hud.rs::paint_version_popup` paints a centered borderless panel (`borderless_panel`, title "What's new in vX — Enter to close") that scales in/out via `popup_scale` (entrance EaseOutCubic/200ms, dismissal EaseInQuad/120ms — easing curves from `pixtuoid_scene::anim`). The clickable "More details" URL's screen rect is `hud.rs::version_popup_url_rect`, derived off the SAME `version_popup_envelope` geometry the painter uses — borderless, so the URL sits flush (`popup_x + URL_PREFIX.len()`, no border inset) and clips against the exclusive envelope edges; keep the two in lockstep (the phantom-browser-launch regression class). (The old `pulse_border_color` border-breathing glow was removed with the border.)
- "How does the `?` help overlay work?" → `tui/widgets/help.rs::paint_help_overlay` paints a centered borderless panel (`borderless_panel`, title "? Keyboard") listing every shortcut (incl. `s` sources). Open state is owned by `UiState.help_open` (toggled by `?`; Enter/Esc/`?` dismiss — the close guard runs BEFORE the version-popup/theme-picker key handlers in `tui/mod.rs`) and mirrored each frame to the renderer's `TuiRenderer::help_open` copy via `set_help_open`. Threaded through `DrawCtx.help_open`, painted last in `draw_scene`.
- "How does the agent dashboard work?" → `Tab` toggles a modal popup: the PURE model `tui::dashboard` + the painter `widgets::dashboard::paint_dashboard`. The model turns `SceneState` into a foldable parent→subagent **tree** (`build_dashboard_rows`: roots by `desk_index`, subagents nested, orphans-as-roots, recursive so deeper nesting is never silently dropped); `DashboardFolds` auto-collapses any root over `AUTO_COLLAPSE_THRESHOLD` (5) so a ~20-subagent CC workflow doesn't flood the board (manual `←/→` fold overrides and STICKS via `user_toggled`; `z` fold-all). Selection is tracked by `AgentId` (`move_selection`/`reanchor_selection` — stable across the 30fps rebuild, re-anchors when the selected agent exits); `Enter` resolves the selected row's floor (`resolve_floor`) and calls `navigate_floor`. State lives in `UiState.dashboard` (persists across open/close for the session); each frame, while open, it builds rows → reanchors → `clamp_scroll` (shared `DASHBOARD_VIEWPORT_ROWS`) → `set_dashboard_frame` to the renderer, which paints the centered/cleared **borderless** popup (via `borderless_panel`) via `DrawCtx.dashboard` (a `&DashboardFrame` bundling `open`/`rows`/`selected`/`scroll`; the row-slice borrow is disjoint from the `floors` (per-floor session halves) borrows; the transition path clones for its brief frames). `paint_dashboard` has ONE call site — the shared `paint_overlays` dispatcher (`renderer.rs`), which both `draw_scene` AND `render_transition` route through (centralized so the two paths can't drift). Dispatch precedence: onboarding > help > version > connection > **dashboard** > theme-picker > normal (mutually exclusive with the picker; `Tab` only opens from the normal scene, and the modal passes the quit chord through). Model unit tests live in `dashboard/tests.rs`; painter/render coverage in `tui_renderer/harness` (`dashboard_*`). **Per-row presentation** (`dashboard_line`): a leading per-CLI `[xx]` badge built by the SHARED `mod.rs::source_badge_span` (the ONE badge builder, also used by the Sources panel + the tooltip dossier; colors from `pixtuoid_scene::theme::Theme::source`/`SourceColors` via `descriptor_for(row.source).label_prefix`, unknown source → `[??]`/`label_idle` fallback); the agent **name is tinted by `RowState`** (active/waiting/idle); and the badge span is intentionally **NOT** `REVERSED` on the selected row (a low-luminance hue inverted vanishes against the highlight bg — only the name/floor/state spans reverse). A `⋮ N more ▾` cue marks below-window overflow, reserving its own line so it never displaces the selected row (and folding back to full height when the selection scrolls to the very end — no blank line). **The SELECTED row's overflowing name + activity cells auto-scroll** (ping-pong): `marquee_or_truncate` swaps the static `…`-`truncate` for `marquee_window` (a stateless `now`-windowed slice — `widgets/mod.rs`, consts `MARQUEE_MS_PER_CHAR`/`MARQUEE_END_PAUSE_MS`); non-selected rows stay `…`-truncated, the scrolling window emits no `…`, and `[p]ause` freezes `now` (and thus the scroll). The painter takes `now` (threaded through the shared `paint_overlays` dispatcher, so both draw paths agree). Badge-color correctness is guarded by `source_badges_legible_for_every_theme` (`pixtuoid_scene::theme::mod.rs`) + `every_registry_source_has_a_non_fallback_badge_color` (`widgets/dashboard.rs`). For headless capture, the `snapshot --dashboard` flag renders a fixed representative scene (a `cc` parent + 2 subagents + a `cx` root + an `rx` root, `dashboard_scene`) with the popup open — driven by the `dashboard` job in `scripts/media.json` → `docs/images/dashboard.png`.
- "How does the footer instrument work?" → `widgets/hud.rs::status_segments` builds `(text, SegRole)` pieces once from a `FooterStats { counts, per_floor, gateway }` (the office-wide truth assembled once per frame at the 3 `paint_footer` sites — `draw_scene`, `draw_footer_only_frame`, `render_transition`); `build_status_spans` (production) tints each by role, `build_status_summary` (`#[cfg(test)]`) concatenates the SAME pieces as the byte-identical text oracle for the insta snapshots. Per-state rungs use the shared `StateKind` vocabulary — `●A ◐W ○I ◌x` glyph+count+letter, **exiting a first-class rung** (the old footer counted all agents while the board counted live-only — now BOTH read `mod.rs::scene_stats`, the anti-drift spine, so they can't disagree mid-walkout). The tool tally tints from the TYPED `Active.kind` via `pixtuoid_scene::pixel_painter::tool_glow_for_kind` (never a name re-parse). Office-wide cues: a right-flushed `▲F{n}` cross-floor waiting cue (from `per_floor_counts`) + the `⬢gw {state}` gateway chip (from `gateway_rollup(scene.daemons())`) — both ALWAYS present (they don't hang off `Option<FloorInfo>`, which is `None` for a single-floor office). Tier fallback (FULL `●3 A · Edit×2 · ⬢gw ok` → MEDIUM compact `●3A ◐2W ○7I` → MIN `▲N · count`, the waiting alarm LEADING) + the right-flush padding key off **display-column** width (`mod.rs::display_width`, unicode-width) — the footer carries single-column multi-byte glyphs (·×↑↓●◐○◌⬢▲), so byte length would short-pad the row. **A source-death warning (#157) preempts the tiers**: when `DrawCtx.source_warning` is set (from the runtime health channel via `TuiRenderer::set_source_warning`), the warning replaces the stats portion entirely (they'd be stale — the office is partially frozen), truncates-with-… instead of tiering away (it must survive every width), keeps the `[q]uit` suffix, and uses `SegRole::Warning` = the `label_waiting` attention color (no new theme key). Painted on both the normal and floor-transition paths. **The `source_warning` string is a TWO-TIER merge** (`doctor::footer_warning(source_death, drifted)` in `run_tui`): source-death (highest — a frozen office) short-circuits, else a passive **decode-drift nudge** (`⚠ decode drift: cc· cx· — run pixtuoid doctor`) when any registered source has accumulated `pixtuoid::drift` breadcrumbs. The drift list comes from `doctor::drifted_sources`, which `run_tui` throttle-scans from the warn-floor log (`RunConfig.log_path`, resolved by `main`) at most every ~15s AND incrementally (`tui/ui_state.rs::DriftScan` (scanned by `UiState::build_frames`): a persistent byte offset — each pass reads only the appended COMPLETE lines and merges prefixes, since breadcrumbs are monotone in-session; rotation is startup-only, so an unbounded in-session log must never be re-read whole inline in the ~30fps loop) — the ONE deliberate exception to "no scan-the-history" (it derives a passive diagnostic hint from the log artifact, NOT lifecycle state; a counting `tracing::Layer` was rejected to avoid stateful blast radius on the single global file subscriber). Drift is breadcrumb-driven ONLY — never the `verified_version` skew, which stays advisory in `doctor`. `None` log path (headless) = no drift surfacing. Breadcrumbs from a prior rotated session (in the `.old` log) are not reflected — both the footer and `doctor` read only the primary log.
- "How does the Sources panel work?" → `s` toggles the **Sources** panel (renamed from `c`/"Connection" once it grew past bind/unbind into a per-source board — connection + live + health; the module + types keep the historical `connection`/`Connection*`/`tui::connection` names, only the UI title + keybind changed): the PURE model `tui::connection` + the painter `widgets::connection::paint_connection_panel`. It is the SOLE way to bind/unbind a source — there is **no `install-hooks`/`uninstall-hooks` CLI** (deleted; `pixtuoid` is the only command). ONE navigable list, one row per agent CLI = the **union** of install targets + registry sources keyed on the source id (`build_rows(&connected_set, log)` joins `source::registry::REGISTRY` to install targets via **`install::target::by_source`** — on `Target.core_source`, NOT `Target.name`, which differs for Claude: target `"claude"` vs source `"claude-code"`; joining on `name` made the flagship row a no-op row, the review CRITICAL pinned by `build_rows_makes_every_source_with_a_target_actionable` + `every_target_names_a_registered_source`; Antigravity has no target → connect/disconnect is a flag-only flip). Each row shows a colored `[xx]` badge (same `pixtuoid_scene::theme::Theme::source` mapping as the dashboard, never `REVERSED`), a **Connection** facet (`ConnState`: `connected`/`disconnected`/`no CLI`) and a **Live** facet (`N agents · age` / `idle` / `⚠ transport died`), plus a socket line and a **state-aware detail line** (armed-confirm prompt > last action result > **health verdict** > per-state: Connected → `installed at: <path>`, Disconnected → `press t to connect` (no path — meaningless until bound), NoCli → `<name> not detected on this machine`). **The health verdict** (the #309 health-consolidation: `⚠ install broken: …` > `⚠ N decode drift — run pixtuoid doctor`) is `doctor::diagnose(src, log).summary()`, cached on `ConnectionRow.health` and computed in `build_rows` for CONNECTED rows ONLY (`connected.then(diagnose…)` — a disconnected row's stale soundness/drift isn't actionable, and `diagnose` itself gates the install check on `has_hooks`, so a just-disconnected source is never falsely "broken"); it PREEMPTS the benign per-state line. `doctor` is the health PROVIDER, `ConnState` the connection lifecycle it ANNOTATES (sub-state, not overlap). Version skew stays `doctor`-only (the probe is too costly for the panel-open). **Two facets, two lifecycles**: the CONNECTION facet is driven by the live connected-set (the persisted `[sources]` intent) + an FS presence read (`is_present` → `NoCli`), so `UiState.connection.rows` is cached — rebuilt on open + after each toggle (`build_rows(&connected.snapshot(), log)`), NEVER per frame; the LIVE facet (`live_view`) is recomputed per frame from the scene snapshot (group `AgentSlot.source`, max `last_event_at`) + the captured `&[SourceDeath]` (the same health snapshot the footer reads — no retained renderer map). **One toggle, not separate i/u**: `t` on a Disconnected row CONNECTS immediately (additive, reversible — `tui::mod::connect_source`: persist the flag via `config::save_source_connected`, `connected.set(src, true)`, then `install::install_target` for target-bearing rows); `t` on a Connected row arms an inline `y/n` confirm → `tui::mod::disconnect_source` (persist false, `connected.set(src, false)`, `install::uninstall_target`), both via the load-bearing `ConfigLock` round; results render through `connection::format_{connect,disconnect}_result`. **The panel's blocking I/O runs under `tokio::task::block_in_place`** — `connect_source`/`disconnect_source` take an advisory `flock` + `fsync`, and `build_rows` does per-source FS probes + `diagnose` config reads; without yielding the worker, that round stalls the input/render tasks under lock contention (census #266 escape #1 — a dropped #283 bot MEDIUM). Valid because `run_tui` runs on the multi-thread runtime (`driver.rs` `new_multi_thread` + inline `block_on`); the sync `connect_source`/`disconnect_source` fns stay wrap-free so their unit tests run outside a runtime. All five panel-I/O sites (open-build + connect + disconnect + their two rebuilds) are wrapped. The property is a scheduling concern at the codecov-ignored event loop, so it's documented here rather than behaviorally tested. The live gate + graceful eviction live in `runtime/driver.rs::reducer_task` (see the binary guide): `connected.set` is the panel's mutation seam, read by the per-event gate (`event_source`) + the per-tick reconciler (`Reducer::reconcile_connected`, which evicts every slot whose source is the COMPLEMENT of the connected-set — so it also sweeps a blank-source gate-slipper), so a disconnect walks characters out live (no restart). Dispatch precedence: onboarding > help > version > **connection** > dashboard > theme-picker > normal; the open tier splits armed (`y`/`n`/`Esc` only) vs unarmed (`j`/`k`/`t`/`s`/`Esc`); the quit chord always passes through. `s` opens/closes it; bare `c` is now UNBOUND (the panel moved off `c`); only `Ctrl+C` is the quit chord. The socket path + the `ConnectedSources` handle are threaded from `runtime/driver.rs` into `run_tui`. Painted in BOTH draw paths via `DrawCtx.connection` (a `&ConnectionFrame` bundling `open`/`rows`/`live`/`selected`/`confirm`/`result`/`socket_line`). **The SELECTED row's name + the (selection-driven) detail line auto-scroll** when they overflow — `marquee_or_truncate`/`marquee_window` (shared with the dashboard, `widgets/mod.rs`) ping-pong the focused text; the detail line is always the selected row's, so it scrolls whenever its `installed at: <path>` overruns the panel. Non-selected rows stay `…`-truncated. The painter takes `now` (same threading as the dashboard). Headless capture: `snapshot --connection` renders a fixed representative fixture; model tests in `connection/tests.rs`, painter tests in `widgets/connection.rs`, render + dispatch tests in `harness`/`mod.rs`. Badge legibility reuses `source_badges_legible_for_every_theme` + the connection twin of `every_registry_source_has_a_non_fallback_badge_color`.
- "How does first-run onboarding work?" → on the FIRST launch ever (`setup::is_first_run` = config absent OR `[sources]` empty — UNLESS the config load DEGRADED (exists-but-malformed/unreadable ⇒ not a first run: onboarding's apply would be refused row-by-row by the malformed-config-never-wiped rule); computed in `main::build_run_config` and threaded `RunConfig.first_run → TuiSession → run_tui`), the TUI plays a one-time "move-in" overlay — the PURE model `tui::welcome::WelcomeUi` (roster of `sources::detect()`'d CLIs, all pre-checked) + the painter `widgets::welcome::paint_welcome` (borderless, a typewriter subtitle + a STAGED roster reveal, all driven by `OnboardingFrame.elapsed_ms` since open — the model holds no clock). The older `connection_*`/`dashboard_*`/`version_popup_*` parallel-field mirrors have since been consolidated into the same shape — `ConnectionFrame`/`DashboardFrame`/`PopupState` on `TuiRenderer` → `DrawCtx.connection`/`.dashboard` — so every render-frame mirror is now one struct. **The overlay is "open" exactly while `UiState`'s `onboarding_opened_at: Option<Instant>` is `Some`** (also the typewriter clock); it stays closed when no CLIs are detected (nothing to connect). **Apply (`Enter`)**: `sources::apply_choices(cfg, &decisions)` connects the checked / disconnects the unchecked, SCOPED to the detected roster — deliberately NOT the plan's declarative `reconcile_to`, which is over EVERY registered source and would write flags for the whole complement incl. `antigravity` (never in `detect()` — since 0.12.0 an unflagged source is plainly disconnected, and an unlisted source's flag should never be written by onboarding); then `connected.set(..)` reflects each into the LIVE gate so the office is ready this session. **Skip (`Esc`)**: persists each detected source's CURRENT state (a freeze) so `[sources]` becomes non-empty (onboarding never re-triggers) yet NO hooks are added/removed (a pre-existing install survives a skip); it then reflects that freeze into the LIVE gate via the SAME `reflect_onboarding_outcomes` the Confirm arm uses — a pre-0.12 upgrader's hooked source (frozen `true`) must go live THIS session, not stay evicted until the next restart re-seeds the gate from the flags. ## When refactoring

The terminal render path is exercised by the headless harness
(`tui_renderer/harness`, ~100 tests — 99 as of #435) driving the real `TuiRenderer` through a
ratatui `TestBackend` — output-first (`buf()` pixels + `frame_buffer` cells).
Changes to `draw_scene`, the widgets, or the dispatch precedence should add or
update a harness test. The engine-side authority (`derive_with_routing`,
`MotionState`, the pixel passes) lives in the `pixtuoid-scene` crate — see
[`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md)'s "When
refactoring." Don't reach back into `floating/` from here: `tui` and `floating`
are sibling painters that share the `pixtuoid-scene` crate, not each other.
