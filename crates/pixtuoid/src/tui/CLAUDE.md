# pixtuoid/tui ‚Äî terminal renderer agent guide

The **terminal painter**: ratatui `App` + `TuiRenderer` (the `Renderer` trait
impl). Owns the half-block flush, the widgets (footer/wall-display/theme-picker
UI/tooltip/help/dashboard/Sources panel), mouse hit-testing, the version popup,
the crossterm event loop + terminal lifecycle, and the per-session UI models
(dashboard, Sources/connection). This is where the `pixtuoid-scene` engine's pixel
buffer becomes terminal cells.

**The render + simulation ENGINE is its OWN crate
[`pixtuoid-scene`](../../../pixtuoid-scene/CLAUDE.md)** (the DAG is
`pixtuoid-core ‚Üê pixtuoid-scene ‚Üê pixtuoid`) ‚Äî layout, pose/motion/pathfinding,
the pixel pass (`render_to_rgb_buffer`), the color theme MODEL, pets, chitchat,
frame_cache, embedded_pack. `tui` and `floating` are sibling thin painters over
the `pixtuoid-scene` crate; neither depends on the other. When a painter entry
below references engine internals (`pixtuoid_scene::pixel_painter`,
`pixtuoid_scene::theme`, `pixtuoid_scene::motion`, ‚Ä¶) follow the link to the
[scene guide](../../../pixtuoid-scene/CLAUDE.md) for the deep "why."

Parent guides: binary [`../../CLAUDE.md`](../../CLAUDE.md); workspace
[`../../../../CLAUDE.md`](../../../../CLAUDE.md); headless lib
[`../../../pixtuoid-core/CLAUDE.md`](../../../pixtuoid-core/CLAUDE.md); the engine
[`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md).

## Known sharp edges (don't be surprised by these)

- **`draw_scene` is called through `TuiRenderer`** (the `Renderer` trait impl), which owns the cross-frame state (RgbBuffer, FrameCache, Router, OccupancyOverlay, PoseHistory) and assembles a per-frame DrawCtx borrow (RgbBuffer, FrameCache, Router, OccupancyOverlay, PoseHistory, theme, mouse state). It delegates the world render to `pixtuoid_scene::pixel_painter::render_to_rgb_buffer` (the shared engine seam ‚Äî see [`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md)), then does the terminal-only half-block flush + widgets + footer. `draw_scene` returns `Result<Option<Layout>>` ‚Äî the computed layout is cached on `TuiRenderer.cached_layout` so hit-test functions can use it without recomputing. During floor transitions, `cached_layout` is cleared to `None`.
- **The version popup's URL click-rect (`version_popup_url_rect`) derives its offsets from the SAME `PANEL_PAD_*` consts the painter insets by** (`url_x = popup_x + PANEL_PAD_X + URL_PREFIX.len()`, edges clipped at `¬±PANEL_PAD_*`) ‚Äî keep them in lockstep (the phantom-browser-launch regression class).
- **Every crossterm key event passes `should_dispatch_key` first** (`tui/mod.rs`): Windows delivers Press AND Release (and Repeat) per keystroke, so only `KeyEventKind::Press` dispatches ‚Äî otherwise toggle keys double-fire there (`p` pauses then instantly unpauses). Unix only emits Press, so the gate is a no-op locally.
- **The `w` walkable/approach/route debug overlay is dev-only** ‚Äî its dispatch arm is `#[cfg(debug_assertions)]`-gated, so release builds silently ignore `w`. Don't "fix" a report that `w` does nothing in an installed binary.

## Where to look

- "How is the office rendered (pixel pass ‚Üí terminal)?" ‚Üí the world render is `pixtuoid_scene::pixel_painter::render_to_rgb_buffer` (see [`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md) for the pixel-pass split). The terminal-flush pass (`half-block + widgets + status footer`) is **`tui::renderer::draw_scene`**. `draw_scene` is the ONE consumer still on raw `render_to_rgb_buffer` (it needs the full `PixelPassResult` + immutable coffee borrows mid-flush); every headless surface goes through the shared `pixtuoid_scene::floor::render_floor` seam (#423) ‚Äî the floor-slide calls it directly (projected scenes, caller-side eviction), while `floating::offscreen::OfficeRenderer` and the web hero each own a `pixtuoid_scene::floor::FloorSession` (the scene-owned session whose `render()` also runs the dual eviction); `TuiRenderer` composes the same session halves as `Vec<PerFloor>` + one `PerOffice`. `examples/snapshot.rs` saves the ratatui `TestBackend` via `save_backend_as_png` (a ‚ñÄ-compressed PNG with footer).
- "How does the neon wall display work?" ‚Üí the pixel-buffer half (dark panel + pulsing cyan border) is `pixtuoid_scene::pixel_painter::background::lighting.rs::paint_neon_panel`; `widgets/hud.rs::paint_wall_display` overlays ratatui text (branding, state dots, scrolling ticker); `widgets/mod.rs::TickerQueue` manages the persistent scrolling message buffer.
- "How does the theme PICKER UI work?" ‚Üí the theme MODEL (the `Theme` struct, `ALL_THEMES`, `SourceColors`/`ApplianceColors`) lives in `pixtuoid_scene::theme` (see [`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md)). The UI is here: press `[t]` for a live preview picker (j/k or ‚Üë‚Üì to navigate); each row shows a 2-cell swatch (`hud.rs::theme_swatch` ‚Üí `neon_brand` + `carpet_base`) drawn from that row's own palette. The picker is dispatched in `tui/mod.rs`; `set_theme()` (on `TuiRenderer`) flushes the `FrameCache` so character recolors update immediately. Theme is saved on `[t]` picker Enter confirm in `tui/mod.rs`.
- "How are popups framed?" ‚Üí **every popup is borderless** (no outline) and shares ONE backing: `widgets/mod.rs::paint_card_backing(f, area, theme)` = a flat single-color drop **shadow** (a hand-rolled fg+bg dim ‚Äî NOT ratatui's `Block::shadow`, which smears over half-blocks) + `Clear` + a solid `tooltip_bg` fill. **The shadow is a deliberately FLAT offset silhouette, not a soft penumbra** (`cast_drop_shadow`): the card's own rect is darkened by ONE uniform `SHADOW_FACTOR` (0.42) and offset `SHADOW_OFFSET` (1) cell down-right; the card paints over its own cells afterward, leaving an L-band (right column offset down, bottom row offset right, meeting at the corner) ‚Äî all ONE color. The **bottom-most row is rendered TOP-HALF only** (`dim_cell`'s `top_half_only`: dims `fg`, leaves `bg` lit) so the bottom band reads as a 1px contact line instead of a full 2px cell, while the vertical right strip stays full cells. This is a pinned user preference (matches a supplied reference ‚Äî flat single color, offset, darker, thin bottom); do NOT "restore" a gradient/falloff or per-band factor, and do NOT make the bottom a full 2px cell ‚Äî the offset silhouette with the 1px top-half bottom is the intended look (`panel::borderless_panel_casts_a_flat_offset_shadow` pins it). Both card kinds delegate to it, so the look (incl. the shadow) has a single definition and can't drift: `tui/widgets/panel.rs::borderless_panel` calls it, then insets by a uniform `PANEL_PAD_X`/`PANEL_PAD_Y` (the breathing room that replaces the border ‚Äî re-exported from `widgets`), draws an optional bold brand-colored title row, and returns the inner content `Rect` (Help / theme-picker / version-popup / dashboard / connection route through it, each `centered_in` adding `2√óPANEL_PAD_*`); the hover/info tooltips (`tooltip.rs::framed_tooltip`, a `Padding::uniform(1)` text-only Block ‚Äî the bg/clear/shadow now come from the backing) call it too. Chitchat speech bubbles are deliberately NOT this primitive (a bare 1-row `Paragraph`, no `Clear`/block) so they get no shadow. The title moved from the old border into the body. **The version popup's URL click-rect (`version_popup_url_rect`) derives its offsets from the SAME `PANEL_PAD_*` consts the painter insets by** (`url_x = popup_x + PANEL_PAD_X + URL_PREFIX.len()`, edges clipped at `¬±PANEL_PAD_*`) ‚Äî keep them in lockstep (the phantom-browser-launch regression class).
- "How does the version popup render?" ‚Üí `hud.rs::paint_version_popup` paints a centered borderless panel (`borderless_panel`, title "What's new in vX ‚Äî Enter to close") that scales in/out via `popup_scale` (entrance EaseOutCubic/200ms, dismissal EaseInQuad/120ms ‚Äî easing curves from `pixtuoid_scene::anim`). The clickable "More details" URL's screen rect is `hud.rs::version_popup_url_rect`, derived off the SAME `version_popup_envelope` geometry the painter uses ‚Äî borderless, so the URL sits flush (`popup_x + URL_PREFIX.len()`, no border inset) and clips against the exclusive envelope edges; keep the two in lockstep (the phantom-browser-launch regression class). (The old `pulse_border_color` border-breathing glow was removed with the border.)
- "How does the `?` help overlay work?" ‚Üí `tui/widgets/help.rs::paint_help_overlay` paints a centered borderless panel (`borderless_panel`, title "? Keyboard") listing every shortcut (incl. `s` sources). Open state on `TuiRenderer::help_open` (toggled by `?`; Enter/Esc/`?` dismiss ‚Äî the close guard runs BEFORE the version-popup/theme-picker key handlers in `tui/mod.rs`). Threaded through `DrawCtx.help_open`, painted last in `draw_scene`.
- "How does the agent dashboard work?" ‚Üí `Tab` toggles a modal popup: the PURE model `tui::dashboard` + the painter `widgets::dashboard::paint_dashboard`. The model turns `SceneState` into a foldable parent‚Üísubagent **tree** (`build_dashboard_rows`: roots by `desk_index`, subagents nested, orphans-as-roots, recursive so deeper nesting is never silently dropped); `DashboardFolds` auto-collapses any root over `AUTO_COLLAPSE_THRESHOLD` (5) so a ~20-subagent CC workflow doesn't flood the board (manual `‚Üê/‚Üí` fold overrides and STICKS via `user_toggled`; `z` fold-all). Selection is tracked by `AgentId` (`move_selection`/`reanchor_selection` ‚Äî stable across the 30fps rebuild, re-anchors when the selected agent exits); `Enter` resolves the selected row's floor (`resolve_floor`) and calls `navigate_floor`. State lives in `run_tui`'s `DashboardUi` (persists across open/close for the session); each frame, while open, it builds rows ‚Üí reanchors ‚Üí `clamp_scroll` (shared `DASHBOARD_VIEWPORT_ROWS`) ‚Üí `set_dashboard_frame` to the renderer, which paints the centered/cleared **borderless** popup (via `borderless_panel`) via `DrawCtx.dashboard` (a `&DashboardFrame` bundling `open`/`rows`/`selected`/`scroll`; the row-slice borrow is disjoint from the `floors` (per-floor session halves) borrows; the transition path clones for its brief frames). `paint_dashboard` has ONE call site ‚Äî the shared `paint_overlays` dispatcher (`renderer.rs`), which both `draw_scene` AND `render_transition` route through (centralized so the two paths can't drift). Dispatch precedence: onboarding > help > version > connection > **dashboard** > theme-picker > normal (mutually exclusive with the picker; `Tab` only opens from the normal scene, and the modal passes the quit chord through). Model unit tests live in `dashboard/tests.rs`; painter/render coverage in `tui_renderer/harness.rs` (`dashboard_*`). **Per-row presentation** (`dashboard_line`): a leading per-CLI `[xx]` badge colored from `pixtuoid_scene::theme::Theme::source` (`SourceColors`), resolved via `descriptor_for(row.source).label_prefix` ‚Üí a `match` arm (unknown source ‚Üí `[??]`/`label_idle` fallback); the agent **name is tinted by `RowState`** (active/waiting/idle); and the badge span is intentionally **NOT** `REVERSED` on the selected row (a low-luminance hue inverted vanishes against the highlight bg ‚Äî only the name/floor/state spans reverse). A `‚ãÆ N more ‚ñæ` cue marks below-window overflow, reserving its own line so it never displaces the selected row (and folding back to full height when the selection scrolls to the very end ‚Äî no blank line). **The SELECTED row's overflowing name + activity cells auto-scroll** (ping-pong): `marquee_or_truncate` swaps the static `‚Ä¶`-`truncate` for `marquee_window` (a stateless `now`-windowed slice, same wallclock trick as `TickerQueue::visible` ‚Äî `widgets/mod.rs`, consts `MARQUEE_MS_PER_CHAR`/`MARQUEE_END_PAUSE_MS`); non-selected rows stay `‚Ä¶`-truncated, the scrolling window emits no `‚Ä¶`, and `[p]ause` freezes `now` (and thus the scroll). The painter takes `now` (threaded through the shared `paint_overlays` dispatcher, so both draw paths agree). Badge-color correctness is guarded by `source_badges_legible_for_every_theme` (`pixtuoid_scene::theme::mod.rs`) + `every_registry_source_has_a_non_fallback_badge_color` (`widgets/dashboard.rs`). For headless capture, the `snapshot --dashboard` flag renders a fixed representative scene (a `cc` parent + 2 subagents + a `cx` root + an `rx` root, `dashboard_scene`) with the popup open ‚Äî driven by the `dashboard` job in `scripts/media.json` ‚Üí `docs/images/dashboard.png`.
- "How does the footer color-code state?" ‚Üí `widgets/hud.rs::status_segments` builds `(text, role)` pieces once; `build_status_spans` (production) tints active/waiting/idle counts by hue (`label_active`/`waiting`/`idle`); `build_status_summary` (`#[cfg(test)]`) concatenates the same pieces as the byte-identical text-contract oracle for the insta snapshots. Tier fallback (full‚Üímedium‚Üímin‚Üíquit-only) and the right-flush padding key off **display-column** width (`chars().count()`, matching the source-warning branch) ‚Äî the footer carries single-column multi-byte glyphs (¬∑, √ó, ‚Üë‚Üì), so byte length over-counts and short-pads the row (was a real bug; the column measure right-flushes `[q]uit` to the exact edge). **A source-death warning (#157) preempts the tiers**: when `DrawCtx.source_warning` is set (from the runtime health channel via `TuiRenderer::set_source_warning`), the warning replaces the stats portion entirely (they'd be stale ‚Äî the office is partially frozen), truncates-with-‚Ä¶ instead of tiering away (it must survive every width), keeps the `[q]uit` suffix, and uses `SegRole::Warning` = the `label_waiting` attention color (no new theme key). Painted on both the normal and floor-transition paths. **The `source_warning` string is a TWO-TIER merge** (`doctor::footer_warning(source_death, drifted)` in `run_tui`): source-death (highest ‚Äî a frozen office) short-circuits, else a passive **decode-drift nudge** (`‚öÝ decode drift: cc¬∑ cx¬∑ ‚Äî run pixtuoid doctor`) when any registered source has accumulated `pixtuoid::drift` breadcrumbs. The drift list comes from `doctor::drifted_sources`, which `run_tui` throttle-scans from the warn-floor log (`RunConfig.log_path`, resolved by `main`) at most every ~15s AND incrementally (`tui/mod.rs::DriftScan`: a persistent byte offset ‚Äî each pass reads only the appended COMPLETE lines and merges prefixes, since breadcrumbs are monotone in-session; rotation is startup-only, so an unbounded in-session log must never be re-read whole inline in the ~30fps loop) ‚Äî the ONE deliberate exception to "no scan-the-history" (it derives a passive diagnostic hint from the log artifact, NOT lifecycle state; a counting `tracing::Layer` was rejected to avoid stateful blast radius on the single global file subscriber). Drift is breadcrumb-driven ONLY ‚Äî never the `verified_version` skew, which stays advisory in `doctor`. `None` log path (headless) = no drift surfacing. Breadcrumbs from a prior rotated session (in the `.old` log) are not reflected ‚Äî both the footer and `doctor` read only the primary log.
- "How does the Sources panel work?" ‚Üí `s` toggles the **Sources** panel (renamed from `c`/"Connection" once it grew past bind/unbind into a per-source board ‚Äî connection + live + health; the module + types keep the historical `connection`/`Connection*`/`tui::connection` names, only the UI title + keybind changed): the PURE model `tui::connection` + the painter `widgets::connection::paint_connection_panel`. It is the SOLE way to bind/unbind a source ‚Äî there is **no `install-hooks`/`uninstall-hooks` CLI** (deleted; `pixtuoid` is the only command). ONE navigable list, one row per agent CLI = the **union** of install targets + registry sources keyed on the source id (`build_rows(&connected_set, log)` joins `source::registry::REGISTRY` to install targets via **`install::target::by_source`** ‚Äî on `Target.core_source`, NOT `Target.name`, which differs for Claude: target `"claude"` vs source `"claude-code"`; joining on `name` made the flagship row a no-op row, the review CRITICAL pinned by `build_rows_makes_every_source_with_a_target_actionable` + `every_target_names_a_registered_source`; Antigravity has no target ‚Üí connect/disconnect is a flag-only flip). Each row shows a colored `[xx]` badge (same `pixtuoid_scene::theme::Theme::source` mapping as the dashboard, never `REVERSED`), a **Connection** facet (`ConnState`: `connected`/`disconnected`/`no CLI`) and a **Live** facet (`N agents ¬∑ age` / `idle` / `‚öÝ transport died`), plus a socket line and a **state-aware detail line** (armed-confirm prompt > last action result > **health verdict** > per-state: Connected ‚Üí `installed at: <path>`, Disconnected ‚Üí `press t to connect` (no path ‚Äî meaningless until bound), NoCli ‚Üí `<name> not detected on this machine`). **The health verdict** (the #309 health-consolidation: `‚öÝ install broken: ‚Ä¶` > `‚öÝ N decode drift ‚Äî run pixtuoid doctor`) is `doctor::diagnose(src, log).summary()`, cached on `ConnectionRow.health` and computed in `build_rows` for CONNECTED rows ONLY (`connected.then(diagnose‚Ä¶)` ‚Äî a disconnected row's stale soundness/drift isn't actionable, and `diagnose` itself gates the install check on `has_hooks`, so a just-disconnected source is never falsely "broken"); it PREEMPTS the benign per-state line. `doctor` is the health PROVIDER, `ConnState` the connection lifecycle it ANNOTATES (sub-state, not overlap). Version skew stays `doctor`-only (the probe is too costly for the panel-open). **Two facets, two lifecycles**: the CONNECTION facet is driven by the live connected-set (the persisted `[sources]` intent) + an FS presence read (`is_present` ‚Üí `NoCli`), so `connection_ui.rows` is cached ‚Äî rebuilt on open + after each toggle (`build_rows(&connected.snapshot(), log)`), NEVER per frame; the LIVE facet (`live_view`) is recomputed per frame from the scene snapshot (group `AgentSlot.source`, max `last_event_at`) + the captured `&[SourceDeath]` (the same health snapshot the footer reads ‚Äî no retained renderer map). **One toggle, not separate i/u**: `t` on a Disconnected row CONNECTS immediately (additive, reversible ‚Äî `tui::mod::connect_source`: persist the flag via `config::save_source_connected`, `connected.set(src, true)`, then `install::install_target` for target-bearing rows); `t` on a Connected row arms an inline `y/n` confirm ‚Üí `tui::mod::disconnect_source` (persist false, `connected.set(src, false)`, `install::uninstall_target`), both via the load-bearing `ConfigLock` round; results render through `connection::format_{connect,disconnect}_result`. **The panel's blocking I/O runs under `tokio::task::block_in_place`** ‚Äî `connect_source`/`disconnect_source` take an advisory `flock` + `fsync`, and `build_rows` does per-source FS probes + `diagnose` config reads; without yielding the worker, that round stalls the input/render tasks under lock contention (census #266 escape #1 ‚Äî a dropped #283 bot MEDIUM). Valid because `run_tui` runs on the multi-thread runtime (`driver.rs` `new_multi_thread` + inline `block_on`); the sync `connect_source`/`disconnect_source` fns stay wrap-free so their unit tests run outside a runtime. All five panel-I/O sites (open-build + connect + disconnect + their two rebuilds) are wrapped. The property is a scheduling concern at the codecov-ignored event loop, so it's documented here rather than behaviorally tested. The live gate + graceful eviction live in `runtime/driver.rs::reducer_task` (see the binary guide): `connected.set` is the panel's mutation seam, read by the per-event gate (`event_source`) + the per-tick reconciler (`Reducer::reconcile_connected`, which evicts every slot whose source is the COMPLEMENT of the connected-set ‚Äî so it also sweeps a blank-source gate-slipper), so a disconnect walks characters out live (no restart). Dispatch precedence: onboarding > help > version > **connection** > dashboard > theme-picker > normal; the open tier splits armed (`y`/`n`/`Esc` only) vs unarmed (`j`/`k`/`t`/`s`/`Esc`); the quit chord always passes through. `s` opens/closes it; bare `c` is now UNBOUND (the panel moved off `c`); only `Ctrl+C` is the quit chord. The socket path + the `ConnectedSources` handle are threaded from `runtime/driver.rs` into `run_tui`. Painted in BOTH draw paths via `DrawCtx.connection` (a `&ConnectionFrame` bundling `open`/`rows`/`live`/`selected`/`confirm`/`result`/`socket_line`). **The SELECTED row's name + the (selection-driven) detail line auto-scroll** when they overflow ‚Äî `marquee_or_truncate`/`marquee_window` (shared with the dashboard, `widgets/mod.rs`) ping-pong the focused text; the detail line is always the selected row's, so it scrolls whenever its `installed at: <path>` overruns the panel. Non-selected rows stay `‚Ä¶`-truncated. The painter takes `now` (same threading as the dashboard). Headless capture: `snapshot --connection` renders a fixed representative fixture; model tests in `connection/tests.rs`, painter tests in `widgets/connection.rs`, render + dispatch tests in `harness.rs`/`mod.rs`. ## When refactoring

The terminal render path is exercised by the headless harness
(`tui_renderer/harness.rs`, ~100 tests ‚Äî 99 as of #435) driving the real `TuiRenderer` through a
ratatui `TestBackend` ‚Äî output-first (`buf()` pixels + `frame_buffer` cells).
Changes to `draw_scene`, the widgets, or the dispatch precedence should add or
update a harness test. The engine-side authority (`derive_with_routing`,
`MotionState`, the pixel passes) lives in the `pixtuoid-scene` crate ‚Äî see
[`../../../pixtuoid-scene/CLAUDE.md`](../../../pixtuoid-scene/CLAUDE.md)'s "When
refactoring." Don't reach back into `floating/` from here: `tui` and `floating`
are sibling painters that share the `pixtuoid-scene` crate, not each other.
