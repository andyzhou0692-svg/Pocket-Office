# pixtuoid-core ÔÇö agent guide

The **headless library**: no terminal dependencies (`ratatui`/`crossterm`/`stdout`
are forbidden here ÔÇö see workspace invariant #1). Owns the source/decoder seam,
the reducer/state machine, sprite parsing, terminal-agnostic layout geometry,
and the pure physics/pose math. The binary (`pixtuoid`) and renderer
(`pixtuoid/src/tui`) sit on top of this. See the workspace
[`CLAUDE.md`](../../CLAUDE.md) for cross-cutting rules.

## Known sharp edges (don't be surprised by these)

- **CC hook payloads DO include `tool_use_id`** in `PreToolUse` and `PostToolUse` (verified by sniffing live payloads). The decoder reads it; the reducer's hook-wins dedup actually fires. **The dedup is asymmetric by event kind (#150)**: a recorded hook End suppresses both JSONL kinds (covers the tool's whole lagged JSONL pair), but a recorded hook Start never suppresses a JSONL End ÔÇö when the PostToolUse hook drops (best-effort shim), that JSONL End is the only completion signal left, and eating a Task self-End would leak `active_tasks` for the rest of the session. The kind lives in the map *value* (an End overwrites its tool's Start entry) ÔÇö don't "simplify" to a kind-in-the-key or exact-kind match; both regress a pinned case (`late_batched_jsonl_pair_after_delivered_hook_end_is_fully_dropped`).
- **CC hook `transcript_path` always points to the PARENT'S transcript**, even when a subagent is the actor ÔÇö so subagent hook events hash to the parent's `AgentId`. The reducer's `active_tasks: HashMap<AgentId, HashSet<String>>` suppresses hook `ActivityStart`/`End` for any agent currently inside a `Task` tool; JSONL has correct subagent attribution via the per-subagent transcript file at `<parent_uuid>/subagents/agent-<id>.jsonl`. The Task signal travels as `ToolDetail::Task` (a typed enum variant on `AgentEvent::ActivityStart.detail`); the reducer pattern-matches on `d.is_task()` rather than scanning a free-form string. **`make_tool_detail` detects the subagent-dispatch tool SEMANTICALLY** ÔÇö by the presence of `subagent_type` in the tool input ÔÇö falling back to the known names (`Agent` in current CC, legacy `Task`). This is deliberate resilience: the dispatch tool was renamed `Task`Ôćĺ`Agent` (CC v2.1.63, **undocumented** ÔÇö issue #29677), silently disabling suppression + b1 completion (the parent then shows the subagent's tools ÔÇö observed live) until we keyed on the stable `subagent_type` field. Don't revert to name-only matching. A dispatch under an unrecognized name still works (semantic) and logs a `debug!` drift breadcrumb so the known-name set / docs can be updated. **Liveness flows UP the tree** (`scope::refresh_lineage`, in the Layer-B `state/scope.rs` module that owns the parentÔćösubagent tree ops): any `ActivityStart`/`ActivityEnd`/`Waiting` refreshes `last_event_at` for the actor AND every ancestor, so a delegating parent isn't stale-swept while a subagent is still working ÔÇö whether the work arrives as a suppressed hook (misattributed to the parent) OR as the subagent's own JSONL (hooks are best-effort; this covers dropped ones). It's the mirror of `cascade_exit` (which pushes EXIT *down*). The suppress block relies on this up-refresh ÔÇö don't drop it, or a single delegation > `STALE_ACTIVE_TIMEOUT` (10 min) would make the live parent look stale and the cascade would evict its still-working subagent.
- **JSONL watcher skips historical transcripts ÔÇö on EVERY first-sight path, not just startup.** `walk_jsonl` in `source/jsonl.rs` runs ONE first-sight gate (`should_seed_at_eof`) the first time *any* pass sees a file (the initial seed, the 250ms rescan, the 60s poll, a notify event): a `.jsonl` with mtime outside `DEFAULT_INITIAL_WINDOW` (currently 1 hour; configurable via `JsonlWatcher::with_initial_window`) **OR** carrying a `session_end`/`SessionEnd` marker (tail-scanned last 8 KB) has its cursor seeded at end-of-file and emits NO `SessionStart`; only a recent, not-ended file is read from the top. Without this, ~hundreds of stale sessions saturate the desk allocator (capacity is auto-computed from terminal size; the headless fallback is `FALLBACK_DESKS=16` in `pixtuoid/runtime/mod.rs`). Long-idle live sessions only re-appear after they next write. A 250ms post-startup rescan catches files missed by `read_dir` during the initial seed (APFS metadata propagation race). **Unifying the gate inside `walk_jsonl` is the #85 fix** ÔÇö it used to be split (only the old `initial_seed_walk` gated; the rescan/poll/notify read first-sight files from offset 0), so when the initial `read_dir` lost the APFS race the 250ms rescan resurrected a missed ended/stale session as a phantom `SessionStart` (a real resurrect-on-restart bug, not only a test flake).
- **Watch backend: native in prod, polling in tests.** `JsonlWatcher::run` watches via `notify`'s `RecommendedWatcher` (FSEvents/inotify/ÔÇŽ) plus the 60s `scan_root` poll backstop. Integration tests call `force_polling_backend_for_tests(interval)` ÔÇö a `#[doc(hidden)]`, set-once process hook ÔÇö to swap in `notify::PollWatcher` at ~25ms instead: a real FSEvents stream costs **tens of seconds per test to set up + tear down** on macOS (it was the bulk of the watcher suite's runtime, and the real-FSEvents coupling was the seam behind the #85 flake ÔÇö see the line above). Both backends impl `notify::Watcher` and feed the SAME channel, so `run`'s `select!` loop is backend-agnostic. The poll uses **`with_compare_contents(true)`** (hash-based) on purpose: `PollWatcher`'s default change-date diff misses rapid appends/truncate-rewrites (the `watcher_does_not_consume_partial_trailing_line` + `watcher_resets_cursor_on_truncation_below_cursor` cases). This seam swaps only the *delivery* mechanism, never the logic ÔÇö the first-sight gate stays covered watcher-free by the deterministic `walk_jsonl` unit tests. Don't wire a real FSEvents watcher into tests.
- **Agent removal needs a `SessionEnd`; abrupt exits have none and fall back to the slow stale-sweep.** A slot is GC'd `EXIT_GRACE_WINDOW` (4.5 s) after `exiting_at` is set, which only an `AgentEvent::SessionEnd` or `sweep_stale` sets. CC emits `SessionEnd` two ways: the **hook** (best-effort ÔÇö 200 ms write timeout, no retry, races CC's own teardown, so it drops silently), and ÔÇö for a **clean `/exit`┬Ě`/quit`** ÔÇö a **durable JSONL marker** decoded by `decode_cc_line` + `cc_session_ended` (CC writes no `session_end` line, only the `<command-name>` user event, so we match that). That JSONL path is the fallback when the hook drops. But **Ctrl-C / terminal-close / kill produce NEITHER** (the process dies before the hook runs and writes no marker), and **Codex has no `SessionEnd` concept at all** (its hooks are `UserPromptSubmit`/`Stop`/`PermissionRequest`; `Stop` = turn end Ôćĺ Idle, not session end). Those exits are reaped only by `sweep_stale` (Active 10 min / Idle 30 min / Waiting 60 min). For **CC** this is largely inherent ÔÇö its clean exits DO signal, so the only CC sessions that reach the long timeout are genuinely live-but-idle or abruptly killed; a short CC silence-reaper would evict legitimately-idle live sessions (lunch-break idle), so don't add one. **Codex gets a carve-out** because it has NO exit signal of ANY kind: no `SessionEnd` hook (its upstream `HookEventName` enum has none ÔÇö only `Stop` = turn end), no PID in any payload, and its internal `EventMsg::ShutdownComplete` is *not persisted* to the rollout (`should_persist_event_msg` Ôćĺ false). All three verified against upstream `openai/codex`. So `sweep_stale` is Codex's ONLY reaper for EVERY exit, and at the 30-min idle timeout an exited `cx┬Ě` sprite ghosts for half an hour. Codex Idle therefore uses the shorter `STALE_CODEX_IDLE_TIMEOUT` (5 min, in `reducer.rs`, selected by the registry-derived `SourceCaps::short_idle_reap()` = `!has_exit_signal && resurrects_on_prompt` ÔÇö Codex is the only row where both hold) ÔÇö safe *specifically* for Codex because the only false-positive (a live-but-idle session past 5 min) is **self-healing**: its next `UserPromptSubmit` re-emits `SessionStart` and the sprite walks back in. CC keeps the 30-min `STALE_IDLE_TIMEOUT` (its real `SessionEnd` signals make a short reaper there all cost, no benefit). Pinned by `codex_idle_agent_reaps_faster_than_claude_idle` in `tests/reducer.rs`. **A stale-swept parent now cascades its whole subagent subtree** via the shared `scope::cascade_exit` (the parentÔćösubagent tree ops live in `state/scope.rs`; same `parent_id` BFS the `SessionEnd` arm uses) ÔÇö so an abrupt-exit parent takes its subagents with it rather than orphaning them on their own longer stale timers. **Two guards stop the cascade from evicting a *live* subagent** (modeled on k8s liveness-vs-readiness): (1) **liveness up-propagation** ÔÇö a working subagent keeps its ancestors fresh (`refresh_lineage`, see the `active_tasks` edge above); (2) **readiness exemption** ÔÇö a subagent blocked under a `Waiting` ancestor (its permission `Notification` was attributed to the parent, so the parent is `Waiting` while the subagent stays `Active`) is skipped by `sweep_stale` via `has_waiting_ancestor` ÔÇö "not ready", not dead. So the stale-sweep cascade only fires once the whole subtree is genuinely silent. **A subagent that COMPLETES is removed promptly too** (no 30-min linger): CC writes no completion marker, so it's inferred ÔÇö when the parent's last `Task` drains (`active_tasks[parent]` empties) the delegated subtree has returned, and `scope::cascade_exit(parent)` marks it exiting (the parent keeps running). **The b1 cascade is grace-DEFERRED, not immediate** (`B1_CASCADE_GRACE`, 2.5s, #151): a parallel second Task dispatch is suppressed as a leak and tracked only via its JSONL copy, so an immediate cascade at the first Task's drain would evict the second's LIVE subtree while that copy is still in watcher latency ÔÇö unrecoverably (`exiting_at` has no clearer; after the GC the child's JSONL events no-op forever). Any Task insert inside the grace cancels the pending cascade (`pending_b1_cascades`); the drain's own tool_result write triggers the notify that replays the backlogged dispatch line, so one FSEvents hop is all the grace must cover. Don't "fix" the ~2.5s completed-subagent linger back to immediacy ÔÇö and note the linger is unbounded for CHAINED dispatches: a sequential re-dispatch inside the grace defuses the pending entry, so the previous wave's completed children wait for the NEXT drain-to-empty (worst case the 30-min idle sweep) ÔÇö the unavoidable price of never evicting a live subtree, since the cascade is whole-subtree. Residuals (consciously accepted, see #151): the chained linger above, and a sustained notify outage spanning the grace (both writes missed Ôćĺ 60s poll) can still evict early. A source with a clean completion signal would drive the same cascade through its own decoder; the scope ops stay source-agnostic.
- **Codex subagents (`spawn_agent`) are wired via the `SubagentStart`/`SubagentStop` HOOKS, not JSONL paths.** Unlike CC (whose subagent has a `<parent>/subagents/agent-*.jsonl` transcript Ôćĺ `detect_parent_id` derives the parent from the path), a Codex subagent's rollout is FLAT (`~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`), so the JSONL watcher renders it but ORPHANED. The hooks are the only carrier of the parent link: `SubagentStart`/`SubagentStop` payloads carry a distinct `agent_id` (the subagent, == its rollout-filename UUID) plus the parent `session_id`. `decode_hook_payload` keys the CHILD on `agent_id` (coalescing with that rollout) and emits `SessionStart{parent_id: session_id}` / `SessionEnd{agent_id}` ÔÇö so the Codex subagent joins the SAME `scope` tree (cascade/liveness/readiness) as a CC one. Because the two transports race, `reducer::apply`'s `SessionStart` arm **enriches** an existing orphan's `parent_id` (rather than no-op-on-exists) when the JSONL slot was created first; it never re-parents an agent that already has a parent. The two `CODEX_EVENTS` registrations (`install/codex.rs`) are therefore live, not dead ÔÇö captured wire format is regression-pinned in `tests/codex_subagent_lifecycle.rs`.
- **Subagent display names come from `attributionAgent` in JSONL.** The decoder strips the plugin prefix (`feature-dev:code-explorer` Ôćĺ `code-explorer`) and emits `AgentEvent::Rename` so labels read meaningfully. Parents get their `cwd` basename instead. `cc_derive_label` falls back to the CC **project dir** (its name encodes the cwd path, `'/'Ôćĺ'-'`) when a Rename's seed line has no `cwd`, so an empty-cwd Rename never degrades `cc┬Ědotfiles` back to a bare `cc`.
- **`AgentSlot.state_started_at` is `std::time::SystemTime`** ÔÇö process-local in practice (no wall-clock anchoring), but the type is already serializable, so the v2 daemon split won't need a type swap. The pose system computes elapsed time relative to it for animation timing.
- **`ActivityState::Active` Ôëá "tool is currently executing".** CC fires PreToolUse Ôćĺ PostToolUse around every individual tool call, so without debouncing the slot flickers Active/Idle on every tool. The reducer treats `ActivityEnd` as "arm pending idle" (`AgentSlot.pending_idle_at`) instead of an immediate flip; the actual transition to `Idle` is realized by `reducer::tick` after `ACTIVE_GRACE_WINDOW = 1500 ms`. Any `ActivityStart` inside the window cancels the pending idle. Net: the slot reads as continuously Active for chained tool work, and visible Idle lags real Idle by up to 1.5 s + the 1 s sweep interval (Ôëł 2.5 s worst case). Don't add code that depends on `Active Ôćĺ Idle` being instant.
- **The reducer's permission `Waiting` resolves on the gated tool's PostToolUse.** A `PermissionRequest`/decision sets `Waiting`; the matching tool's `ActivityEnd` (PostToolUse) clears it (`gated_before_waiting` tracking). It is also cleared by `SessionEnd` + retained-eviction; `gated_before_waiting` is evicted at BOTH sites ÔÇö `tick`'s `retain` and `sweep_exited`'s explicit `remove` (the apply-path eviction, pinned by `gated_before_waiting_evicted_on_apply_path_sweep`). A THIRD clear site lives on the Task-drain path (#152), CONDITIONAL on the drained tuid matching the gate: the drain deliberately skips the main arm (a Waiting parent keeps Waiting), so its gate entry would go stale and a late out-of-window JSONL replay of the same END would false-resolve a still-pending permission. The conditionality is load-bearing ÔÇö an unconditional clear would wipe a parallel ordinary tool's gate (pinned by `task_drain_keeps_parallel_ordinary_tool_gate`). Keep the sites in sync if you touch eviction.
- **A meeting room narrower than `MEETING_FURNITURE_MIN_W` (compute.rs) has NO sofa/table/seats ÔÇö bare floor, BY DESIGN.** Below it the 16px sofa body leaves too little margin for the coarse 4├Ś4 router to reach the seats, so an idle agent sent to sit would TELEPORT (find_path None Ôćĺ straight-line fallback). The room Bounds/walls/door still exist; only the unroutable furniture is dropped (same degradation the dense floor uses when too short). So "meeting room exists" no longer implies "meeting slots exist" ÔÇö don't "fix" the missing sofa at small sizes. Guarded by `pathfind::tests::every_wander_waypoint_is_routable_on_the_coarse_grid` (coarse-grid reachability across seeds ├Ś sizes ÔÇö stronger than the pixel-BFS connectivity sweep).
- **Occlusion is EMERGENT ÔÇö there is no `occludes_behind` field / synthetic cap any more (deleted).** An overhanging piece (`visual.h > footprint.h`: plant canopy, booth column, TV monitor, whiteboard panel, pantry counter) has its footprint **south-anchored to the sprite base** by `mask::stamp_overhang_aware`, so a walker parks DEEP behind it and the piece's own y-sorted sprite paints over their lower body. The old `paint_furniture_back_floor` floor-snapshot cap is gone. **Two intentionally decoupled footprints:** the MASK uses the shallow `footprint` (occlusion + collision); `stand_point`/`approach_point` derive the USER's park distance from the full `visual` (`half_extents`/`approach_footprint`) so someone *using* the furniture parks clear of the whole sprite, not inside it. This is the industry Y-sort pattern (base-collision + sprite-occludes); **don't re-add a cap** ÔÇö make the sprite overhang its shallow footprint instead. Pinned by `every_pod_occludes_via_overhang`.
- **Pantry counter blocks only a shallow `PANTRY_FOOTPRINT_DEPTH` south strip, not its full sprite height.** The counter is a ┬ż-view sprite centered on `pos`; only its south base contacts the floor (the receding cabinet tops overhang, invariant #6). `mask.rs` south-anchors a 3px strip; a character behind it is occluded by the counter's own y-sorted sprite, couch-style ÔÇö the same emergent occlusion as every overhanging piece (above). `stand_point` parks the USER clear of the full `visual`, not this shallow strip (the runtime-sized pantry was the original mask-vs-approach decouple, now generalised to all overhanging obstacles).

## Where to look

- "Where does a furniture's footprint / visual size / approach side / dwell come from?" Ôćĺ **`core::layout::decor`** is the single source of truth, and there is now exactly ONE geometry table. `FurnitureDef { footprint, visual, occupies_pos, dwell, approach: ApproachSides }` + `furniture_def(Furniture)` ÔÇö keyed by the unified `Furniture` geometry enum ÔÇö plus `desk_furniture_def()` (the home desk: same `FurnitureDef` type, NOT a `Furniture`/waypoint row, since it's owned per-agent). **The role enums carry NO dimensions**: `WaypointKind` (wander destination), `PodDecor` (aisle filler), `PlantKind`, `WallDecor` each expose `.furniture()` mapping onto one `Furniture` row, so overlapping items (a phone booth that is both a waypoint and pod decor, a plant that is both free-standing and pod decor, a whiteboard that is both wall and pod decor) have their shape defined exactly once. The singleton / per-room furniture (meeting sofa-body + table, pantry table/stool, lamp, side table) are ALSO rows now ÔÇö `Furniture::{MeetingSofaBody, MeetingTable, PantryTable, PantryChair, FloorLamp, LoungeSideTable}` ÔÇö distinct from the `MeetingSofa`/`MeetingStand` SEAT rows (3 seats sit on 1 body, so the seats carry `None` footprint and the body carries the obstacle). The ONLY geometry living outside the table is `PLANT_FOOTPRINT` (the one value shared by the four plant rows, named once to avoid duplication) and the linear wall thickness `WALL_THICK_H/V` (walls are a thickness, not a point-centred rect). **`footprint` is the BLOCKED ground area** (z=0 projection) ÔÇö `mask::build_walkable_mask` stamps it (no inline literals); **`visual` is the sprite size** ÔÇö render centering + the z-sort south row (`center_pin_south_offset`) derive from it. **Three load-bearing distinctions:** (1) `footprint` (blocked ground) Ôëá `visual` (sprite, may overhang ÔÇö `Furniture::PlantTall` footprint 6├Ś**3** (a shallow south-anchored pot strip) vs visual 6├Ś10; overhanging pieces have a footprint *shorter* than the sprite so the canopy/column occludes a walker behind it via z-sort; vertical wall footprint 1px vs 3px drawn); (2) `occupies_pos` = the agent sits/stands ON the cell (couch/meeting) vs approaches it from a side (appliances); (3) **placement overlap** (furniture buried in a wall = bug, see `freestanding_decor_does_not_overlap_room_walls`) Ôëá **occlusion overlap** (glass-over-sofa, sofa-over-sitter, canopy-over-walker = physically correct, resolved by the tui z-sort, NOT a placement error). `ApproachSides` (canonical facing-South, rotated by `allows`) is the editable entry-side knob ÔÇö flip one bool to drop a side. The unifier is "one geometry table (footprint + visual) + one shared z-sort," not a second `size()` per role enum.
- "How is the office laid out?" Ôćĺ `core::layout::SceneLayout::compute_with_seed` is the coordinator; it calls 4 private helpers: `compute_pod_desks` (pod grid), `compute_pod_decor` (aisle decor), `compute_room_walls` (wall segments + door gaps), `compute_waypoints` (all waypoint assembly). Re-exported as `tui::layout::Layout`. `core::pose::derive` for pure stateÔćĺpose mapping including the Idle wander state machine (`WANDER_CYCLE_BASE_MS=7000` + per-agent jitter).
- "How does the per-agent state machine work?" Ôćĺ **two stacked layers, both extracted into named modules so the reducer reads as a coordinator.** **Layer A** = `state/fsm.rs`: the single-slot transitions (`enter_active`/`enter_delegating`/`enter_waiting`/`arm_pending_idle`/`settle_to_idle`/`mark_exiting`/`rename`), each independently unit-tested; `accumulate_active_ms` lives here once (was copy-pasted 4├Ś). **Layer B** = `state/scope.rs`: the parentÔćösubagent tree ops. **Invariant:** `fsm.rs` mutates ONE slot; the **cross-slot correlation** (`active_tasks` suppression, `gated_before_waiting` permission, the hook-wins dedup) stays in `reducer.rs` ÔÇö it spans slots and is NOT the FSM. The reducer decides *whether/which* transition fires; `fsm` decides *how* the slot changes. Don't move the correlation maps onto `AgentSlot`.
- "Why is the subagent's sprite the right one and not the parent?" Ôćĺ `reducer::Reducer::apply` does subagent-leak suppression via `active_tasks` before applying. `claude_code::decode_cc_line` emits `AgentEvent::Rename` from `attributionAgent`.
- "How does multi-source decoding work?" Ôćĺ `JsonlWatcher` takes `LineDecoder` and `LabelDeriver` fn pointers (Strategy pattern via fn pointers, not traits). CC wires `claude_code::decode_cc_line` + `cc_derive_label`; Codex wires `codex::decode_codex_line` + its label fn; Antigravity wires `antigravity::decode_ag_line` + `derive_ag_label`. `decoder.rs` holds shared utilities (`describe_tool_target`, `make_tool_detail`, `decode_hook_payload`). Each source owns its own JSONL format knowledge. `AgentId::from_parts(source, path)` namespaces IDs per source. Labels show source prefix: `cc┬Ěproject` / `cx┬Ěproject` / `ag┬Ěproject`. **CLI attribution comes ONLY from the shim-owned `_pixtuoid_source` key, NEVER the public `source` field** (CC overloads `source` for the SessionStart *reason* ÔÇö startup/resume/clear ÔÇö and reading it splits the agent into an un-reapable ghost; see the WHY at `source/decoder.rs` `decode_hook_payload`). Per-source CROSS-CUTTING facts (label prefix, hook id-key/custom decoder, reducer capability flags, the harness's line decoder) live in ONE `SourceDescriptor` row in `source/registry.rs` ÔÇö the reducer's `source_label_prefix`/`stale_threshold` and the fixture harness read the table instead of name-matching, and `SourceCaps::short_idle_reap()` derives the Codex short-idle policy from wire facts (`!has_exit_signal && resurrects_on_prompt` ÔÇö Antigravity lacks the signal too but can't resurrect, hence its long window). Adding a CLI = (1) the source module (SOURCE_NAME + decoders), (2) ONE descriptor row in `registry.rs`, (3) the name in `source::REGISTERED_SOURCES` (a bridge test pins tableÔćölist equality; the fixture conformance test forces a fixture under `tests/fixtures/sources/<name>/`), (4) wire it into `pixtuoid/runtime/driver.rs::run_async` (the registry gates conformance tests, NOT runtime spawning). You do NOT separately edit `source_label_prefix`, `stale_threshold`, or the harness's `decoder_for` ÔÇö those read the table. Ship a fixture exercising the **SessionStart hook** so `tests/fixture_harness.rs`'s one-AgentId assertion guards against the ghost.
- "Why don't old idle sessions show on startup?" Ôćĺ `source::jsonl::should_seed_at_eof` (the first-sight gate inside `walk_jsonl`). Checks `check_session_ended` (tail-scans last 8 KB for `session_end`/`SessionEnd` markers) and seeds the cursor at EOF for any file with mtime > `DEFAULT_INITIAL_WINDOW` (1 hour) ÔÇö no `SessionStart`. An mtime outside the 1-hour window **OR** a `session_end` marker is the freshness gate, and it runs on every first-sight path (seed / 250ms rescan / 60s poll / notify), not only startup.
- "How does walk-pace physics work?" Ôćĺ `pixtuoid_core::physics` (pure, no terminal deps) ÔÇö `WalkIntent` tags the walk kind (Entry/Exit/WanderOut/WanderBack/SnapBack); `walk_profile(path_len_octile, intent, agent_id)` returns a frozen `WalkProfile` with trapezoidal/triangular kinematics (triangular when path < `v┬▓/a`, trapezoidal otherwise); `walk_progress(p, elapsed_ms)` emits `t_x1000 Ôłł [0,1000]`; `walk_arrived` gates the WalkingÔćĺSeated/AtWaypoint flip including the per-agent arrival pause. Constants: `V_CRUISE_COMMUTE = 0.36` octile/ms (Entry/Exit), `V_CRUISE_SNAPBACK = 0.65` octile/ms (SnapBack ÔÇö the urgent IdleÔćĺActive return rushes back), `V_CRUISE_WANDER = 0.25` octile/ms (amble, wander legs), `WALK_ACCEL = 6.5e-4` octile/ms┬▓ (~0.55 s ramp; `L_crit Ôëł 199` octile commute, `Ôëł 96` octile wander), `WALK_ACCEL_SNAPBACK = 2.0e-3` octile/ms┬▓ (~3├Ś ÔÇö short snap-backs are accel-limited, so the rush comes from a harder accelerator, not cruise). Per-agent personality: `speed_mult` (bits 24..34 of splitmix64 hash, range 0.85ÔÇô1.20) and `pause_ms_for` (bits 40..52, range 200ÔÇô400 ms) ÔÇö disjoint from each other and from `cycle_ms_for`/`personality_for`. Near desks arrive and sit before far desks (natural staggered arrivals). The *stateful* timeline that consumes these profiles lives in `pixtuoid/src/tui/motion` (the render authority) ÔÇö see that nested guide.

## Keeping the decode mapping current (upstream drift)

Tool names, hook-event names and transcript types are upstream wire contracts that change **without notice** (the `Task`Ôćĺ`Agent` rename shipped undocumented). Four defenses, in order of reliability:

1. **Resilient detection (in code).** Prefer a *semantic* signal over a hardcoded name where one exists ÔÇö e.g. `make_tool_detail` keys subagent dispatch on the stable `subagent_type` input field, not the tool name. This survives a rename by construction.
2. **Self-monitoring from the real stream (in code).** The decoders already see ground truth. An unrecognized `hook_event_name` `bail!`s + `warn!`s; a dispatch under an unknown name logs a `debug!` breadcrumb. These surface drift the moment it appears in a user's own usage ÔÇö no network, no upstream-layout dependency.
3. **Internal consistency test.** `every_registered_codex_event_decodes` / `every_registered_cc_event_decodes` (in `install/{codex,claude}.rs`) assert that every hook event we *register* has a decoder arm ÔÇö catching the *registered-but-not-decoded* class (the original SubagentStop bug) at `cargo test` time.
4. **Upstream watch (CI).** `scripts/check_upstream_drift.py` + `.github/workflows/upstream-drift.yml` (weekly) read the names we depend on directly from our source and compare to live upstream; on a vanished name or a new unhandled Codex hook it opens a deduped `upstream-drift` issue. Sources:
   - **Codex** (open Rust): `codex-rs/protocol/src/protocol.rs` `HookEventName` enum (raw on `main`). `PreCompact`/`PostCompact` are intentionally omitted (`CODEX_KNOWN_OMITTED` in the script) ÔÇö not agent activity.
   - **CC** (closed binary ÔÇö no source to grep): the docs markdown `code.claude.com/docs/en/tools-reference.md` (authoritative tool-name list). Tool names like `Agent` are NOT schematized upstream, so defense #2 is the real backstop for CC.
