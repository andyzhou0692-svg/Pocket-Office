# pixtuoid-core — agent guide

The **headless library**: no terminal dependencies (`ratatui`/`crossterm`/`stdout`
are forbidden here — see workspace invariant #1). Owns the source/decoder seam,
the reducer/state machine, sprite parsing, and the shared grid/walkability
vocabulary. The sim geometry (layout, pose derivation, walk physics) lives in
`pixtuoid-scene` — the engine owns its geometry; only the coherence-bound
`walkable.rs` stays here (see sharp edges). The scene engine (`pixtuoid-scene`)
and the binary (`pixtuoid`) sit on top of this. See the workspace
[`CLAUDE.md`](../../CLAUDE.md) for cross-cutting rules.

## Known sharp edges (don't be surprised by these)

- **`render/` is now just the `test-renderer` fixture home — the legacy `Renderer` trait was RETIRED (#483).** The pre-scene-split render layer was swept in 0.12.0 (blit helpers `blit_frame_outlined`/`draw_line`/`draw_dotted_hline`/`half_block_cells`/`HalfCell`, `sprite/animator`) — only `blit_frame` stays (the scene painter's primitive). The `Renderer` trait's two impls rode it non-polymorphically (`TuiRenderer` prod flush + `TestRenderer` e2e), so #483 inlined both to inherent `render` methods and deleted the trait + its `lib.rs` re-export. New render targets go through `pixtuoid_scene::floor::render_floor` / `pixel_painter::render_to_rgb_buffer` (workspace invariant #1) — do NOT reintroduce a core render trait.
- **`walkable.rs` is coherence-bound to this crate — it did NOT move with the sim-geometry cluster.** When layout/pose/physics moved to `pixtuoid-scene` (the engine owns its geometry), `WalkableMask` had to stay: it is a type ALIAS for `Grid<bool>` whose obstacle ops (`new_open`/`mark_blocked`/`mark_walkable`/`is_walkable`) are an inherent `impl Grid<bool>`, and Rust's orphan rule pins an inherent impl to the crate that owns the type — `Grid` is core vocabulary (`Frame`/`RgbBuffer` newtype it). Reversing the #333 alias back to a wrapper struct, or an extension trait, were both already adjudicated against (see the comment on the impl). So core keeps the mask VOCABULARY; the producers (`pixtuoid_scene::layout::mask`) and consumers (pathfind/pose/motion) live crate-up.
- **The `native` (default) feature gates the ASYNC SOURCE RUNTIME, not the decoders — and the gates are MODULE-level, not item scatter.** `default-features = false` drops tokio/notify and everything behind `#[cfg(feature = "native")]`: `hook/`, `jsonl/`, `manager`, the liveness probes (`cc_probe`/`fd_probe`/`exit_watch`), `source/native.rs` (the `Source`/`DynSource` traits + tagged channel, ONE gate in `source/mod.rs`), and each mixed source module's runtime half — its `impl Source`, watcher wiring, probe attach, `PresenceExitWatch` — which lives in a once-gated `native` sub-module (`source/<cli>/native.rs` for claude_code/codex/antigravity/copilot; `source/daemon/native.rs` for the presence layer), re-exported from the parent so public paths don't move. That leaves the pure decode/reducer/sprite core that compiles to `wasm32-unknown-unknown` (what `pixtuoid-scene` → the `pixtuoid-web` painter build against). **Per-source decoder fns and the registry stay always-compiled on purpose** (registry rows name them as data; `LineDecoder` + `SUBAGENTS_DIR` + `is_subagent_path` live in the always-compiled `decoder.rs` for the same reason — `jsonl` re-exports, never redefines). When gating new code, follow the pattern: put the whole native half in the module's `native.rs` behind the parent's ONE gate + re-export (a parent file should carry ≤2 item-level `cfg`s — the gate and the re-export), never item-by-item `#[cfg]` scatter. Label-deriver / session-ended fns go NATIVE-side unless an external consumer needs them `pub` in the parent (CC/Codex/Copilot keep theirs parent-`pub` for tests + `examples/snapshot/`; AG's are native-private — consumer-driven, not arbitrary). `just hack` (feature powerset) catches ungated optional-dep use on the HOST target, but wasm32-only breakage (target-gated deps, syscalls outside the gate) surfaces ONLY in CI's `wasm-check` job / a local `just gen-wasm` — host-green ≠ wasm-green.
- **CC hook payloads DO include `tool_use_id`** in `PreToolUse` and `PostToolUse` (verified by sniffing live payloads). The decoder reads it; the reducer's hook-wins dedup actually fires. **The dedup is asymmetric by event kind (#150)**: a recorded hook End suppresses both JSONL kinds (covers the tool's whole lagged JSONL pair), but a recorded hook Start never suppresses a JSONL End — when the PostToolUse hook drops (best-effort shim), that JSONL End is the only completion signal left, and eating a Task self-End would leak `active_tasks` for the rest of the session. The kind lives in the map *value* (an End overwrites its tool's Start entry) — don't "simplify" to a kind-in-the-key or exact-kind match; both regress a pinned case (`late_batched_jsonl_pair_after_delivered_hook_end_is_fully_dropped`).
- **CC now keys on the session UUID, not the transcript path.** The hook uses `IdKey::SessionId` (the `session_id` field); the watcher overrides `IdDeriver` with `cc_id_from_path` (the transcript filename stem, which equals that UUID); `decode_cc_line` re-derives the same UUID via `cc_id_from_path`; and `detect_parent_id` returns the `<parent-uuid>` directory component — the path segment immediately before `subagents/`. That component is the parent's own `cc_id_from_path` value, so the subagent→parent link is **cwd-split-safe**: even when the subagent transcript lands under a different project dir (a git-worktree cwd-split), only the cwd-derived prefix differs — the `<parent-uuid>` segment is identical. **Antigravity** (its keying) and CC's `SubagentStop` arm (the `agent_transcript_path` stem) fold a path through `id::normalize_path_key` (it lives in `id.rs` with the identity types, shared by the pixtuoid-scene palette's cwd key; identity on Unix; on Windows it strips a `\\?\`/`\\?\UNC\` verbatim prefix then folds `\`→`/` + casefolds — Antigravity emits backslash paths on Windows, NTFS is case-insensitive, and the `\\?\` strip (#197) is defensive so a `canonicalize`-style verbatim path still coalesces with its plain form). `AgentId::from_transcript_path` (the test/example shim) normalizes via that fn too, so test expectations stay platform-consistent by construction. Pinned by `tests/sources/decode/mod.rs::hook_and_watcher_keys_coalesce_for_one_file` (+ cfg(windows) twins). Don't add a CC keying site without routing it through `cc_id_from_path`; don't add an Antigravity site without `normalize_path_key`.
- **CC hook `transcript_path` always points to the PARENT'S transcript**, even when a subagent is the actor — so subagent TOOL hook events hash to the parent's `AgentId`. The reducer's `active_tasks: HashMap<AgentId, HashSet<String>>` suppresses hook `ActivityStart`/`End` for any agent currently inside a `Task` tool; JSONL has correct subagent attribution via the per-subagent transcript file at `<parent_uuid>/subagents/agent-<id>.jsonl`. The subagent→parent link is derived by `detect_parent_id` as the **`<parent-uuid>` directory component** — the path segment immediately before `subagents/` — which equals `cc_id_from_path` of the parent transcript; this is cwd-split-safe (a git-worktree subagent transcript lands under a different project-dir prefix, but the `<parent-uuid>` segment is identical). **Since #241 CC also decodes its OWN `SubagentStart`/`SubagentStop` hooks** (`claude_code::decode_cc_hook_custom`, dispatched via the registry's `hook.custom` exactly like Codex's — these change the event's SUBJECT to the child, inexpressible in the shared session-keyed arms): `SubagentStart` registers the child instantly with the parent link (the JSONL watcher's later `SessionStart` for the same transcript coalesces); `SubagentStop` emits the child's `SessionEnd` — the ONLY end signal a **Workflow-fleet** subagent gets (no per-agent `Agent` tool_use in the parent transcript → b1 structurally can't fire; no transcript end marker → finished fleet agents used to hold desks until the stale sweeps). **Keying nuance**: the wire `agent_id` is BARE hex (CC v2.1.170 capture) while the transcript filename stem — the watcher's id space — is `agent-<id>`; Start prefixes the wire id (tolerating an already-prefixed form without double-prefixing), Stop keys on `cc_id_from_path(agent_transcript_path)` (the authoritative key — exact watcher parity, covers the deeper `subagents/workflows/wf_*/` nesting) with the prefixed wire id as fallback. Captured wire fixture + cross-transport lifecycle regression: `tests/sources/claude/`. The Task signal travels as `ToolDetail::Task` (a typed enum variant on `AgentEvent::ActivityStart.detail`); the reducer pattern-matches on `d.is_task()` rather than scanning a free-form string. **`make_tool_detail` detects the subagent-dispatch tool SEMANTICALLY** — by the presence of `subagent_type` in the tool input — THE mechanism, with the single known name `Agent` (current CC) as the fallback for the rare input-less call. This is deliberate resilience: the dispatch tool was renamed `Task`→`Agent` (CC v2.1.63, **undocumented** — issue #29677), silently disabling suppression + b1 completion (the parent then shows the subagent's tools — observed live) until we keyed on the stable `subagent_type` field. Don't revert to name-only matching. **The legacy `Task` name arm was DROPPED in 0.12.0** (pre-v2.1.63 CC is too old to keep supporting): an old CC's real dispatch always carries `subagent_type`, so it stays caught semantically — only a name-only, input-less `Task` call lost coverage (accepted), and a `Task` dispatch WITH the field now fires the `unknown_dispatch` breadcrumb, correctly (an unrecognized dispatch name in the wild IS drift-worthy). A dispatch under any unrecognized name still works (semantic) and logs that `unknown_dispatch` drift breadcrumb (`source/drift.rs`, `warn!` on `pixtuoid::drift`) so the known-name set / docs can be updated. **Liveness flows UP the tree** (`scope::refresh_lineage`, in the Layer-B `state/scope.rs` module that owns the parent↔subagent tree ops): any `ActivityStart`/`ActivityEnd`/`Waiting` refreshes `last_event_at` for the actor AND every ancestor, so a delegating parent isn't stale-swept while a subagent is still working — whether the work arrives as a suppressed hook (misattributed to the parent) OR as the subagent's own JSONL (hooks are best-effort; this covers dropped ones). It's the mirror of `cascade_exit` (which pushes EXIT *down*). The suppress block relies on this up-refresh — don't drop it, or a single delegation > `STALE_ACTIVE_TIMEOUT` (10 min) would make the live parent look stale and the cascade would evict its still-working subagent.
- **JSONL watcher skips historical transcripts — on EVERY first-sight path, not just startup.** `walk_jsonl` in `source/jsonl/walk.rs` runs ONE first-sight gate (`should_seed_at_eof`) the first time *any* pass sees a file (the initial seed, the 250ms rescan, the 60s poll, a notify event): a `.jsonl` with mtime outside `DEFAULT_INITIAL_WINDOW` (currently 1 hour; configurable via `JsonlWatcher::with_initial_window`) **OR** carrying a `session_end`/`SessionEnd` marker (tail-scanned last 8 KB) has its cursor seeded at end-of-file and emits NO `SessionStart`; only a recent, not-ended file is read from the top. Symlinked ENTRIES under the root are refused wholesale before the gate (`walk_jsonl` stats via `symlink_metadata` — a planted dir symlink would recurse a loop / walk foreign trees into this source's id space; the root itself may still be a symlink, only entries are checked). Without this, ~hundreds of stale sessions saturate the desk allocator (capacity is auto-computed from terminal size; the headless fallback is `FALLBACK_DESKS=16` in `pixtuoid/runtime/mod.rs`). mtime is only a liveness PROXY, so the gate composes with an optional ADDITIVE-ONLY **liveness probe** (`JsonlWatcher::with_liveness_probe`, refreshed once per scan pass; notify walks reuse the latest snapshot): a transcript whose `IdDeriver`-derived session id the probe vouches for bypasses the gate — CC wires `claude_code::live_cc_session_ids` (the first-party `~/.claude/sessions/<pid>.json` registry, entries verified live via `kill(pid, 0)` PLUS a PID-reuse identity check (#220): when the entry's `startedAt` (ms epoch) and the kernel start time (`pid_start_time_secs` — macOS `proc_pidinfo(PROC_PIDTBSDINFO)` → `pbi_start_tvsec`) are BOTH available they must agree within 10s (measured live: CC stamps `startedAt` ~1.2s after process start) or the entry is skipped as recycled; either side missing → pid-alive-only, so the check is additive (Linux start time is ticks-since-boot — conversion deferred, identity check macOS-only); registry-shape drift now warns once (#247): an entry that parses as JSON but lacks a consumed key (`pid`/`sessionId` missing or mistyped — `RegistryParse::ShapeDrift`) yields no vouch and emits ONE `tracing::warn!` per process run naming the file + key (the registry is undocumented with no fetchable upstream text to diff, so the consumer is the drift detector; not-JSON files AND value-level corruption (pid <= 0, empty sessionId) stay silent — half-written mid-write transients / junk, not format drift, and the inline `live_shape_entry` fixture pins the current shape); Unix only — Windows returns an empty set = pure mtime), attached only when the projects root sits in the standard `<claude_home>/projects` layout (a `--projects-root /tmp/fixture` replay keeps pure-mtime behavior). Codex wires `codex::live_codex_rollout_ids` (open-FD binding — Codex has NO registry, but a live `codex` process holds its rollout `.jsonl` open in append mode for the whole session, upstream `RolloutRecorder`; pid → open fd → rollout path → UUID via `source/fd_probe.rs`: macOS libproc FFI / Linux `/proc/<pid>/{comm,fd}`; other platforms return the empty set = pure mtime), attached only under codex's first-party layout (`codex_probe_root`: file_name `sessions` AND parent `.codex` OR the resolved `codex_home()/sessions`, so a `CODEX_HOME` user keeps the ladder — a `--codex-sessions-root /tmp/fixture` replay keeps pure-mtime behavior). Subagent transcripts never match the CC probe (their stems are agent ids, not session UUIDs). Other long-idle live sessions (Antigravity — no probe) only re-appear after they next write. **An OVERSIZED span (> `MAX_PENDING_BYTES` = 1 MiB pending) is skipped to EOF, never replayed — but it still registers (#204, bounded head read for cwd/label) and, since #222, tail-scans the last 256 KiB (`TASK_SCAN_BYTES`, `scan_pending_tasks`) for UNMATCHED Task dispatches (an `ActivityStart{detail: Task}` with no later `ActivityEnd` for the same tool_use_id) and re-emits exactly those, Jsonl-tagged** — restoring subagent-leak suppression + b1 completion for a mid-attach to a delegating parent whose in-flight `Agent` dispatch sat in the skipped backlog (its PreToolUse hook predates attach, so `active_tasks` had no other seed). Everything else the scan decodes is DISCARDED (Task-seeding, not a replay), the scan is skipped when the span ENDED (terminator wins) or the file stays unregistered (no slot → JSONL no-ops), and tail-window geometry means a synthesized start can't be a false leak (a completion is always later in the file than its start). Residuals: a dispatch buried deeper than 256 KiB before EOF keeps the old skip behavior (bounded, pinned by `oversized_attach_dispatch_outside_window_is_missed`), and a mid-RUN oversized skip's synthesized start may be dedup-eaten by a recent hook record — correct, the hook copy already seeded `active_tasks` (#150). **The probe is ALSO ongoing liveness, not just admission (#220):** after each refresh (initial seed / 250ms rescan / 60s poll — the three sites that re-snapshot `live`) the watcher emits `AgentEvent::ProofOfLife` for every vouched id (`emit_proof_of_life`, after `scan_root` so fresh admissions already have slots); the reducer keeps a PRIVATE `recent_proof_of_life` map (deliberately not an `AgentSlot` field — no semver surface) with a 150s TTL (`PROOF_OF_LIFE_TTL`, 2.5× the 60s poll: two missed polls + slack) that `sweep_stale`'s candidate pass consults — a vouched slot is never staleness-swept while the process lives (motivating case: a permission-parked CC session renders Active after attach-replay — its hook-only Waiting is unreconstructable from JSONL — and `STALE_ACTIVE_TIMEOUT` would reap it mid-decision). **The vouch also shields the vouched slot's actively-DELEGATING subtree**: the probe never vouches subagent ids (their stems are `agent-<id>`, not session UUIDs) and a permission-parked parent renders Active, not Waiting (so `has_waiting_ancestor` can't exempt its blocked-but-live child) — `sweep_stale` therefore also skips a slot when any ANCESTOR (`scope::has_ancestor_where`, the cycle-guarded walk `has_waiting_ancestor` shares) holds a fresh vouch (`vouch_fresh`, the ONE TTL predicate) AND a non-empty `active_tasks` entry; the active-delegation gate preserves the 30-min idle backstop for completed lingering children (the b1 chained-dispatch residual). When the live signal disappears (registry entry removed / rollout fd closed) emissions stop and normal sweeps resume after the TTL lapse — subtree shield included. **And since #223 the probe drives the NEGATIVE vouch — vouched-then-gone is a high-confidence exit.** Probe failure is now EXPLICIT: `LivenessProbe` returns `Option<ProbeSnapshot>` (`ids` + `pid_of`, the id→owning-pid binding for the exit watch). `None` = the enumeration itself FAILED and the watcher changes NOTHING (admission keeps the previous snapshot, miss windows neither open nor age, no `ProofOfLife` — the reducer's TTL absorbs the gap); `Some` with empty `ids` = a healthy "nothing alive" observation (meaningful!), so absence of an id only counts inside a `Some` snapshot. A previously-vouched id MISSING from two healthy snapshots ≥ `NEGATIVE_VOUCH_MIN_SPAN` (60s — immune to Codex's brief drop-and-reopen fd gap on a write failure and to initial-seed/250ms-rescan adjacency) apart gets a Jsonl-tagged `SessionEnd` — the exit signal the CLI never writes. For Codex an fd close IS session end (including a thread switch within one process: the old rollout closes, the new one opens and gets its own vouch); CC's clean exits already `SessionEnd` via the hook, so this catches the abrupt ones (Ctrl-C, terminal close, kill) in ~60–120s instead of the 10–30 min stale-sweep. CC emits `SessionEnd` ONE way: the **hook** (best-effort — 200 ms write timeout, no retry, races CC's own teardown, so it drops silently); a dropped hook falls back to the idle sweep. There is NO durable JSONL fallback: modern CC persists no `/exit` record at all (slash commands are `type:"system",subtype:"local_command"` lines, and no transcript ends with an /exit user line — 135-transcript corpus), and the old content-based `<command-name>/exit</command-name>` matcher false-positived on messages QUOTING the wrapper, so it was removed — user-controllable content must never drive lifecycle (`cc_session_ended` now matches only structural `session_end`/`SessionEnd` markers, which CC doesn't write today; the upstream-drift watch flags it if that changes). **Ctrl-C / terminal-close / kill produce no signal either** (the process dies before the hook runs), and **Codex has no `SessionEnd` concept at all** — its 10-variant `HookEventName` enum (verified vs `openai/codex` main, 2026-06-13) has no session-end/exit hook; the terminal-ish ones are `Stop` = turn end → Idle and `SubagentStop` = subagent end, neither is session end. (It DOES fire `SessionStart`/`PreToolUse`/`PostToolUse` — see the multi-source-decoding note — but firing is version-unstable, so they're enrichment, not the lifecycle authority.) Those exits are reaped only by `sweep_stale` (Active 10 min / Idle 30 min / Waiting 60 min) — except where the liveness probe runs (macOS/Linux standard layouts): since #223 its NEGATIVE vouch (see the watcher edge above) ends a vouched-then-gone session in ~60–120s, leaving `sweep_stale` as the fallback for the probe's blind spots. For **CC** this is largely inherent — its clean exits usually signal via the hook, so the CC sessions that reach the long timeout are genuinely live-but-idle, abruptly killed, or the rare dropped-hook clean exit; a short CC silence-reaper would evict legitimately-idle live sessions (lunch-break idle), so don't add one. **Codex gets a carve-out** because it has NO exit signal of ANY kind: no `SessionEnd` hook (its upstream `HookEventName` enum has none — its terminal hooks `Stop`/`SubagentStop` are turn/subagent end, not session end), no PID in any payload, and its internal `EventMsg::ShutdownComplete` is *not persisted* to the rollout (`should_persist_event_msg` → false). Pinned by `codex_idle_agent_reaps_faster_than_claude_idle` in `tests/reducer/liveness.rs`. **A stale-swept parent now cascades its whole subagent subtree** via the shared `scope::cascade_exit` (the parent↔subagent tree ops live in `state/scope.rs`; same `parent_id` BFS the `SessionEnd` arm uses) — so an abrupt-exit parent takes its subagents with it rather than orphaning them on their own longer stale timers. **Two guards stop the cascade from evicting a *live* subagent** (modeled on k8s liveness-vs-readiness): (1) **liveness up-propagation** — a working subagent keeps its ancestors fresh (`refresh_lineage`, see the `active_tasks` edge above); (2) **readiness exemption** — a subagent blocked under a `Waiting` ancestor (its permission `Notification` was attributed to the parent, so the parent is `Waiting` while the subagent stays `Active`) is skipped by `sweep_stale` via `has_waiting_ancestor` — "not ready", not dead. So the stale-sweep cascade only fires once the whole subtree is genuinely silent. **A subagent that COMPLETES is removed promptly too** (no 30-min linger), via a two-rung ladder: (1) **the `SubagentStop` hook (#241)** — CC fires it per finished subagent with `agent_transcript_path`, decoding to the child's own `SessionEnd` (instant, and the ONLY end signal for Workflow-fleet subagents, which have no per-agent `Agent` dispatch for b1 to drain); (2) **b1 Task-drain inference** for Agent-tool dispatches when that best-effort hook drops — CC writes no completion marker, so it's inferred: when the parent's last `Task` drains (`active_tasks[parent]` empties) the delegated subtree has returned, and `scope::cascade_exit(parent)` marks it exiting (the parent keeps running); the stale sweeps remain the backstop below both. **The b1 cascade is grace-DEFERRED, not immediate** (`B1_CASCADE_GRACE`, 2.5s, #151): a parallel second Task dispatch is suppressed as a leak and tracked only via its JSONL copy, so an immediate cascade at the first Task's drain would evict the second's LIVE subtree while that copy is still in watcher latency — unrecoverably (`exiting_at` has no clearer; after the GC the child's JSONL events no-op forever). Any Task insert inside the grace cancels the pending cascade (`pending_b1_cascades`); the drain's own tool_result write triggers the notify that replays the backlogged dispatch line, so one FSEvents hop is all the grace must cover. Don't "fix" the ~2.5s completed-subagent linger back to immediacy — and note the linger is unbounded for CHAINED dispatches: a sequential re-dispatch inside the grace defuses the pending entry, so the previous wave's completed children wait for the NEXT drain-to-empty (worst case the 30-min idle sweep) — the unavoidable price of never evicting a live subtree, since the cascade is whole-subtree. Residuals (consciously accepted, see #151): the chained linger above, and a sustained notify outage spanning the grace (both writes missed → 60s poll) can still evict early. A source with a clean completion signal would drive the same cascade through its own decoder; the scope ops stay source-agnostic.
- **Resurrect-in-place starts from clean correlation state.** A `SessionStart` on an EXITING root slot cancels the walkout (`fsm::resurrect_in_place`) AND evicts the dead life's `active_tasks`/`gated_before_waiting`/`pending_b1_cascades` entries (mirroring `sweep_exited`, which never ran because the corpse never GC'd — a leftover tuid can never drain, so it would suppress every hook event of the new life); the `recent_proof_of_life` vouch deliberately survives (the resurrecting process is alive by definition).
- **Codex subagents (`spawn_agent`) are wired via the `SubagentStart`/`SubagentStop` HOOKS, not JSONL paths.** Unlike CC (whose subagent has a `<parent>/subagents/agent-*.jsonl` transcript → `detect_parent_id` derives the parent from the path; CC additionally decodes its own Subagent hooks since #241 — see the transcript_path edge above), a Codex subagent's rollout is FLAT (`~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`), so the JSONL watcher renders it but ORPHANED. The hooks are the only carrier of the parent link: `SubagentStart`/`SubagentStop` payloads carry a distinct `agent_id` (the subagent, == its rollout-filename UUID) plus the parent `session_id`. `decode_hook_payload` keys the CHILD on `agent_id` (coalescing with that rollout) and emits `SessionStart{parent_id: session_id}` / `SessionEnd{agent_id}` — so the Codex subagent joins the SAME `scope` tree (cascade/liveness/readiness) as a CC one. Because the two transports race, `reducer::apply`'s `SessionStart` arm **enriches** an existing orphan's `parent_id` (rather than no-op-on-exists) when the JSONL slot was created first; it never re-parents an agent that already has a parent, and it **refuses any link that would close a `parent_id` cycle** (`scope::would_create_cycle`, checked at this ONE seam for both registration and enrichment — warn + degrade to parentless, #238): with cycles unable to exist, the scope walks need cycle guards only for termination, never correctness (a mutual-Waiting 2-cycle would otherwise exempt both members from `sweep_stale` forever). A MULTI-TURN Codex child (parent `send_input`) gets SubagentStop at EVERY turn end but no new SubagentStart — and NO other SessionStart carrier at turn N+1 from upstream (hook_runtime.rs verified 2026-06-11: UserPromptSubmit fires only for direct user input, non-Subagent events in a child's context carry the ROOT session_id). The carrier is manufactured by the **child-end un-claim side-channel (#246)** — the `SubagentStop`'s decoded `SessionEnd{as_child:true}` is observed by the CC hook tee and the Codex watcher then RELEASES the rollout's `seen` claim (drain-pending-bytes-first, no SessionEnd emitted, `seen`→`false` so the FD probe's re-vouch can't replay the still-open rollout) — so the turn-N+1 append first-sights a parentless SessionStart that the child-ledger re-link (see the hooks-proof-of-life edge above) re-parents; the same adoption covers the other parentless starts: a flat-rollout first-sight (#244-w1) and a post-exit revival. The two `CODEX_EVENTS` registrations (`install/codex.rs`) are therefore live, not dead — captured wire format is regression-pinned in `tests/sources/codex/mod.rs`.
- **Subagent display names come from `attributionAgent` in JSONL.** The decoder strips the plugin prefix (`feature-dev:code-explorer` → `code-explorer`) and emits `AgentEvent::Rename` so labels read meaningfully. Parents get their `cwd` basename instead. `cc_derive_label` falls back to the CC **project dir** (its name encodes the cwd path, `'/'→'-'`) when a Rename's seed line has no `cwd`, so an empty-cwd Rename never degrades `cc·dotfiles` back to a bare `cc`.
- **`AgentSlot.state_started_at` is `std::time::SystemTime`** — process-local in practice (no wall-clock anchoring). The pose system computes elapsed time relative to it for animation timing. **The whole `SceneState` tree derives `Serialize`/`Deserialize` (#279)**: the `Arc<str>`/`Arc<Path>` slot fields cross through owned `String`/`PathBuf` via the local `*_serde` adapters in `state/mod.rs` (serde has no blanket `Arc<T>` impl, and its `rc` feature wouldn't cover `Arc<Path>` — no `Box<Path>: Deserialize`). This exists for debug state dumps + the full-scene regression golden (`tests/reducer/snapshot.rs`, insta) — the serialized shape is NOT a stable wire contract, so adding a slot field is non-breaking (the golden just surfaces it for review). A v2 daemon snapshot/restore would build on this, but that consumer (and the multi-client #280 / remote-probe #281 prereqs) is deferred — closed as out-of-scope for the single-window, embedded-frontend direction.
- **`ActivityState::Active` ≠ "tool is currently executing".** CC fires PreToolUse → PostToolUse around every individual tool call, so without debouncing the slot flickers Active/Idle on every tool. The reducer treats `ActivityEnd` as "arm pending idle" (`AgentSlot.pending_idle_at`) instead of an immediate flip; the actual transition to `Idle` is realized by `reducer::tick` after `ACTIVE_GRACE_WINDOW = 1500 ms`. Any `ActivityStart` inside the window cancels the pending idle. Net: the slot reads as continuously Active for chained tool work, and visible Idle lags real Idle by up to 1.5 s + the 1 s sweep interval (≈ 2.5 s worst case). Don't add code that depends on `Active → Idle` being instant.
- **The reducer's permission `Waiting` resolves on the gated tool's PostToolUse.** A `PermissionRequest`/decision sets `Waiting`; the matching tool's `ActivityEnd` (PostToolUse) clears it (`gated_before_waiting` tracking). It is also cleared by `SessionEnd` + retained-eviction; `gated_before_waiting` is evicted at BOTH sites — `tick`'s `retain` and `sweep_exited`'s explicit `remove` (the apply-path eviction, pinned by `gated_before_waiting_evicted_on_apply_path_sweep`). A THIRD clear site lives on the Task-drain path (#152), CONDITIONAL on the drained tuid matching the gate: the drain deliberately skips the main arm (a Waiting parent keeps Waiting), so its gate entry would go stale and a late out-of-window JSONL replay of the same END would false-resolve a still-pending permission. The conditionality is load-bearing — an unconditional clear would wipe a parallel ordinary tool's gate (pinned by `task_drain_keeps_parallel_ordinary_tool_gate`). A FOURTH conditional site: the suppress-restore path only removes the gate when it holds a Task tuid (∈ `active_tasks`) — an ordinary tuid keeps Waiting + the gate (pinned by `suppressed_child_event_keeps_parents_own_parallel_tool_gate`). Keep the sites in sync if you touch eviction.

## Keeping the decode mapping current (upstream drift)

Tool names, hook-event names and transcript types are upstream wire contracts that change **without notice** (the `Task`→`Agent` rename shipped undocumented). Four defenses, in order of reliability:

1. **Resilient detection (in code).** Prefer a *semantic* signal over a hardcoded name where one exists — e.g. `make_tool_detail` keys subagent dispatch on the stable `subagent_type` input field, not the tool name. This survives a rename by construction.
2. **Self-monitoring from the real stream (in code).** The decoders already see ground truth — and the breadcrumbs are now STRUCTURED: `source/drift.rs` (always-compiled) owns the four decode-drift breadcrumbs (`unknown_event`/`missing_field`/`unknown_dispatch`/`shape_drift`), each a `tracing::warn!` on the stable target `pixtuoid::drift`, so the warn-floor log and `pixtuoid doctor` surface them. An unrecognized `hook_event_name` → `unknown_event`; a dispatch under an unknown name → `unknown_dispatch`; the `sessions/<pid>.json` registry consumer's once-per-run shape warn (#247 — the canonical instance of this defense for a NON-FETCHABLE surface defense #4 can't reach) → `shape_drift`. These surface drift the moment it appears in a user's own usage — no network, no upstream-layout dependency.
3. **Internal consistency test.** `every_registered_codex_event_decodes` / `every_registered_cc_event_decodes` (in `install/{codex,claude}.rs`) assert that every hook event we *register* has a decoder arm — catching the *registered-but-not-decoded* class (the original SubagentStop bug) at `cargo test` time.
4. **Upstream watch (CI).** `scripts/check_upstream_drift.py` + `.github/workflows/upstream-drift.yml` (weekly) read the names we depend on directly from our source and compare to live upstream; on a vanished name or a new unhandled Codex/Reasonix hook it opens a deduped `upstream-drift` issue. **Coverage is both EVENT/TYPE names AND payload FIELD names** — a renamed field is the same silent-failure class (the decoder reads `None`, the sprite quietly breaks), so beyond the event lists below the watch also checks the FIELD names each decoder reads wherever the upstream owner is fetchable: Reasonix payload json tags, Codex `EventMsg`/`ResponseItem` types + `FunctionCall` fields, CodeWhale `DEEPSEEK_*` env vars, opencode schema Struct fields, Copilot schema `properties`, OpenClaw hook-types fields, Hermes `_serialize_payload` keys (`agent/shell_hooks.py`). Each is ONE-DIRECTIONAL (only a VANISHED depended field alarms) and the field SETS are hardcoded literals next to the code (maintained by review, like `REASONIX_PAYLOAD_FIELDS` — a decoder scrape would drag in the defensive `KEYS` arrays + fixture JSON). **When you add/remove a wire field the decoder reads, update that source's `*_PAYLOAD_FIELDS` set** — the set is trusted to equal the decoder's reads (a stray member false-alarms on a field we don't depend on; a missing member silently loses coverage). Cursor + CC (closed binaries, docs-prose) and Antigravity (no fetchable schema) can't be field-watched — defense #2 is their limit. Sources:
   - **Codex** (open Rust): `codex-rs/protocol/src/protocol.rs` `HookEventName` enum (raw on `main`). `PreCompact`/`PostCompact` are intentionally omitted (`CODEX_KNOWN_OMITTED` in the script) — not agent activity. **Plus the ROLLOUT decode vocabulary** (#500): the transcript decoder's `("event_msg"|"response_item", …)` arms are registered NOWHERE and its `_ => vec![]` fall-through drops an unknown type SILENTLY (no `unknown_event` breadcrumb, unlike the hook decoders), so `read_codex_rollout_types` diffs each depended type against the `EventMsg` enum (protocol.rs) + the `ResponseItem` enum (`codex-rs/protocol/src/models.rs`) — one-directional (only a VANISHED depended type alarms; codex emits ~40 EventMsg / ~15 ResponseItem types we ignore). `task_started`/`task_complete` count via their `#[serde(alias="turn_started"/"turn_complete")]`.
   - **CC** (closed binary — no source to grep): the docs markdown `code.claude.com/docs/en/tools-reference.md` (authoritative tool-name list) **and `hooks.md`'s hook-event summary table** — diffed against `install/claude.rs` `EVENTS` (now incl. SubagentStart/SubagentStop, #241) + the script's `CC_KNOWN_OMITTED` set, so a registered event vanishing is breaking drift and a NEW upstream event is a review ping. Tool names like `Agent` are NOT schematized upstream, so defense #2 is the real backstop for CC.
   - **Reasonix** (open Go): `internal/hook/hook.go` `Event` consts + the payload json tags (`event`/`cwd`/`toolName`/`toolArgs`/`subject`/`message`) on `main-v2`. `PostLLMCall`/`PreCompact`/`SubagentStop` are intentionally omitted (`REASONIX_KNOWN_OMITTED`).
   - **CodeWhale** (open Rust): `crates/tui/src/hooks.rs` `pub enum HookEvent` (snake_case wire names) on `main`, diffed against `install/codewhale.rs` `CODEWHALE_EVENTS` + `CODEWHALE_KNOWN_OMITTED`.
   - **opencode** (open TS): the `EventV2` `type` literals in `packages/schema/src/v1/session.ts` + `packages/schema/src/permission.ts` on `dev` (the `main` branch was retired and the schema moved out of `packages/core/`, #406), diffed against the `match event` block in `source/opencode.rs`. ONE-DIRECTIONAL (opencode emits ~50 types, we map a handful) — only a VANISHED depended type alarms; `OPENCODE_TOLERATED` skips the defensive `permission.asked` alias.
   - **Copilot** (published JSON schema): the per-event `type` consts in the per-platform `@github/copilot-<os>-<arch>` (e.g. `copilot-linux-x64`) `schemas/session-events.schema.json` (served by unpkg, UNPINNED — follows the redirect to latest; the root `@github/copilot` is now a loader stub that 404s the schema, #406), diffed against the `match kind` block in `source/copilot.rs`. ONE-DIRECTIONAL like opencode (Copilot emits ~100 types, we map ~10) — only a VANISHED depended type alarms, so there is no `COPILOT_KNOWN_OMITTED` set.
   - **Cursor** (closed binary): the camelCase `hook_event_name`s on `cursor.com/docs/hooks`, diffed against `CURSOR_EVENTS` in `install/cursor.rs`. ONE-DIRECTIONAL (Cursor exposes ~18 hook events, we map ~5) — only a VANISHED depended event alarms; the common-word `stop` is low-confidence (the docs page contains the word regardless), so the distinctive `sessionStart`/`sessionEnd`/`preToolUse`/`postToolUse` carry the check.
   - **OpenClaw** (open TS): the hook-event types in `src/plugins/hook-types.ts` on `openclaw/openclaw` `main`, diffed against `OPENCLAW_EVENTS` in `install/openclaw.rs`. ONE-DIRECTIONAL (~40 upstream types, we register 6) — only a VANISHED registered event alarms.
   - **Hermes** (open Python): the shell-hook event KEYS of `_DEFAULT_PAYLOADS` in `hermes_cli/hooks.py` on `NousResearch/hermes-agent` `main` (the `hermes hooks test`/`doctor` fixtures, so a rename moves the key), diffed against `HERMES_EVENTS` in `install/hermes.rs`. ONE-DIRECTIONAL (~15 upstream shell-hook events, we register 4) — only a VANISHED registered event alarms.
   - **Antigravity** is deliberately **defense-#2-ONLY** — NO drift-watch row (defense #4). It's a closed Google IDE, and while its on-disk `transcript.jsonl` wire IS partially observable (a Google Cloud Community tutorial + the confirmed on-disk path `~/.gemini/antigravity-cli/brain/<id>/.system_generated/logs/` quote real lines: field set `{step_index, source, type, status, created_at, content, tool_calls}`, step types incl. `USER_INPUT`/`PLANNER_RESPONSE`/`SEARCH_WEB`), that is illustrative SAMPLE wire, not a stable published schema — so it is NOT a reliable `check_upstream_drift.py` source (same NON-FETCHABLE-SCHEMA class as the CC `sessions/<pid>.json` registry, #247; do NOT add a reader — there is nothing stable to fetch). Only `query` (the `search_web` arg) is capture-confirmed; the rest of `decode_ag_line`'s tool-arg vocabulary (the PascalCase `DirectoryPath`/`AbsolutePath`/`TargetFile`/`CommandLine`/`SearchPath` keys) and the permission tool names (`ask_permission`/`ask_question`) are reverse-engineered/UNVERIFIED (commented as such in the source). So currency rests on in-code breadcrumbs: `decode_ag_line` fires `drift::missing_field` on a tool-call missing its `name`, AND `drift::unknown_event` when a step carries `tool_calls` under a type that isn't `PLANNER_RESPONSE` (the rename signal — see the source). **Two-file layout:** the CLI writes BOTH `transcript.jsonl` (truncated) and `transcript_full.jsonl` (untruncated) per conversation in one logs dir with the SAME step stream, so `AntigravitySource` sets `JsonlWatcher::with_path_filter(skip_transcript_full)` to watch only the canonical `transcript.jsonl` — else one conversation double-renders as two path-keyed `AgentId`s.
