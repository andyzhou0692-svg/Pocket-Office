# pixtuoid-core — agent guide

The **headless library**: no terminal dependencies (`ratatui`/`crossterm`/`stdout`
are forbidden here — see workspace invariant #1). Owns the source/decoder seam,
the reducer/state machine, sprite parsing, terminal-agnostic layout geometry,
and the pure physics/pose math. The binary (`pixtuoid`) and renderer
(`pixtuoid/src/tui`) sit on top of this. See the workspace
[`CLAUDE.md`](../../CLAUDE.md) for cross-cutting rules.

## Known sharp edges (don't be surprised by these)

- **CC hook payloads DO include `tool_use_id`** in `PreToolUse` and `PostToolUse` (verified by sniffing live payloads). The decoder reads it; the reducer's hook-wins dedup actually fires. **The dedup is asymmetric by event kind (#150)**: a recorded hook End suppresses both JSONL kinds (covers the tool's whole lagged JSONL pair), but a recorded hook Start never suppresses a JSONL End — when the PostToolUse hook drops (best-effort shim), that JSONL End is the only completion signal left, and eating a Task self-End would leak `active_tasks` for the rest of the session. The kind lives in the map *value* (an End overwrites its tool's Start entry) — don't "simplify" to a kind-in-the-key or exact-kind match; both regress a pinned case (`late_batched_jsonl_pair_after_delivered_hook_end_is_fully_dropped`).
- **CC now keys on the session UUID, not the transcript path.** The hook uses `IdKey::SessionId` (the `session_id` field); the watcher overrides `IdDeriver` with `cc_id_from_path` (the transcript filename stem, which equals that UUID); `decode_cc_line` re-derives the same UUID via `cc_id_from_path`; and `detect_parent_id` returns the `<parent-uuid>` directory component — the path segment immediately before `subagents/`. That component is the parent's own `cc_id_from_path` value, so the subagent→parent link is **cwd-split-safe**: even when the subagent transcript lands under a different project dir (a git-worktree cwd-split), only the cwd-derived prefix differs — the `<parent-uuid>` segment is identical. **Only Antigravity** still folds a path through `decoder::normalize_path_key` (identity on Unix; on Windows it strips a `\\?\`/`\\?\UNC\` verbatim prefix then folds `\`→`/` + casefolds — Antigravity emits backslash paths on Windows, NTFS is case-insensitive, and the `\\?\` strip (#197) is defensive so a `canonicalize`-style verbatim path still coalesces with its plain form). `AgentId::from_transcript_path` (the test/example shim) normalizes via that fn too, so test expectations stay platform-consistent by construction. Pinned by `tests/sources/decode/mod.rs::hook_and_watcher_keys_coalesce_for_one_file` (+ cfg(windows) twins). Don't add a CC keying site without routing it through `cc_id_from_path`; don't add an Antigravity site without `normalize_path_key`.
- **CC hook `transcript_path` always points to the PARENT'S transcript**, even when a subagent is the actor — so subagent hook events hash to the parent's `AgentId`. The reducer's `active_tasks: HashMap<AgentId, HashSet<String>>` suppresses hook `ActivityStart`/`End` for any agent currently inside a `Task` tool; JSONL has correct subagent attribution via the per-subagent transcript file at `<parent_uuid>/subagents/agent-<id>.jsonl`. The subagent→parent link is derived by `detect_parent_id` as the **`<parent-uuid>` directory component** — the path segment immediately before `subagents/` — which equals `cc_id_from_path` of the parent transcript; this is cwd-split-safe (a git-worktree subagent transcript lands under a different project-dir prefix, but the `<parent-uuid>` segment is identical). The Task signal travels as `ToolDetail::Task` (a typed enum variant on `AgentEvent::ActivityStart.detail`); the reducer pattern-matches on `d.is_task()` rather than scanning a free-form string. **`make_tool_detail` detects the subagent-dispatch tool SEMANTICALLY** — by the presence of `subagent_type` in the tool input — falling back to the known names (`Agent` in current CC, legacy `Task`). This is deliberate resilience: the dispatch tool was renamed `Task`→`Agent` (CC v2.1.63, **undocumented** — issue #29677), silently disabling suppression + b1 completion (the parent then shows the subagent's tools — observed live) until we keyed on the stable `subagent_type` field. Don't revert to name-only matching. A dispatch under an unrecognized name still works (semantic) and logs a `debug!` drift breadcrumb so the known-name set / docs can be updated. **Liveness flows UP the tree** (`scope::refresh_lineage`, in the Layer-B `state/scope.rs` module that owns the parent↔subagent tree ops): any `ActivityStart`/`ActivityEnd`/`Waiting` refreshes `last_event_at` for the actor AND every ancestor, so a delegating parent isn't stale-swept while a subagent is still working — whether the work arrives as a suppressed hook (misattributed to the parent) OR as the subagent's own JSONL (hooks are best-effort; this covers dropped ones). It's the mirror of `cascade_exit` (which pushes EXIT *down*). The suppress block relies on this up-refresh — don't drop it, or a single delegation > `STALE_ACTIVE_TIMEOUT` (10 min) would make the live parent look stale and the cascade would evict its still-working subagent.
- **JSONL watcher skips historical transcripts — on EVERY first-sight path, not just startup.** `walk_jsonl` in `source/jsonl.rs` runs ONE first-sight gate (`should_seed_at_eof`) the first time *any* pass sees a file (the initial seed, the 250ms rescan, the 60s poll, a notify event): a `.jsonl` with mtime outside `DEFAULT_INITIAL_WINDOW` (currently 1 hour; configurable via `JsonlWatcher::with_initial_window`) **OR** carrying a `session_end`/`SessionEnd` marker (tail-scanned last 8 KB) has its cursor seeded at end-of-file and emits NO `SessionStart`; only a recent, not-ended file is read from the top. Without this, ~hundreds of stale sessions saturate the desk allocator (capacity is auto-computed from terminal size; the headless fallback is `FALLBACK_DESKS=16` in `pixtuoid/runtime/mod.rs`). mtime is only a liveness PROXY, so the gate composes with an optional ADDITIVE-ONLY **liveness probe** (`JsonlWatcher::with_liveness_probe`, refreshed once per scan pass; notify walks reuse the latest snapshot): a transcript whose `IdDeriver`-derived session id the probe vouches for bypasses the gate — CC wires `claude_code::live_cc_session_ids` (the first-party `~/.claude/sessions/<pid>.json` registry, entries verified live via `kill(pid, 0)` PLUS a PID-reuse identity check (#220): when the entry's `startedAt` (ms epoch) and the kernel start time (`pid_start_time_secs` — macOS `proc_pidinfo(PROC_PIDTBSDINFO)` → `pbi_start_tvsec`) are BOTH available they must agree within 10s (measured live: CC stamps `startedAt` ~1.2s after process start) or the entry is skipped as recycled; either side missing → pid-alive-only, so the check is additive (Linux start time is ticks-since-boot — conversion deferred, identity check macOS-only); Unix only — Windows returns an empty set = pure mtime), attached only when the projects root sits in the standard `<claude_home>/projects` layout (a `--projects-root /tmp/fixture` replay keeps pure-mtime behavior). Codex wires `codex::live_codex_rollout_ids` (open-FD binding — Codex has NO registry, but a live `codex` process holds its rollout `.jsonl` open in append mode for the whole session, upstream `RolloutRecorder`; pid → open fd → rollout path → UUID via `source/fd_probe.rs`: macOS libproc FFI / Linux `/proc/<pid>/{comm,fd}`; other platforms return the empty set = pure mtime), attached only under codex's first-party layout (`codex_probe_root`: file_name `sessions` AND parent `.codex` OR the resolved `codex_home()/sessions`, so a `CODEX_HOME` user keeps the ladder — a `--codex-sessions-root /tmp/fixture` replay keeps pure-mtime behavior). Subagent transcripts never match the CC probe (their stems are agent ids, not session UUIDs). Other long-idle live sessions (Antigravity — no probe) only re-appear after they next write. **An OVERSIZED span (> `MAX_PENDING_BYTES` = 1 MiB pending) is skipped to EOF, never replayed — but it still registers (#204, bounded head read for cwd/label) and, since #222, tail-scans the last 256 KiB (`TASK_SCAN_BYTES`, `scan_pending_tasks`) for UNMATCHED Task dispatches (an `ActivityStart{detail: Task}` with no later `ActivityEnd` for the same tool_use_id) and re-emits exactly those, Jsonl-tagged** — restoring subagent-leak suppression + b1 completion for a mid-attach to a delegating parent whose in-flight `Agent` dispatch sat in the skipped backlog (its PreToolUse hook predates attach, so `active_tasks` had no other seed). Everything else the scan decodes is DISCARDED (Task-seeding, not a replay), the scan is skipped when the span ENDED (terminator wins) or the file stays unregistered (no slot → JSONL no-ops), and tail-window geometry means a synthesized start can't be a false leak (a completion is always later in the file than its start). Residuals: a dispatch buried deeper than 256 KiB before EOF keeps the old skip behavior (bounded, pinned by `oversized_attach_dispatch_outside_window_is_missed`), and a mid-RUN oversized skip's synthesized start may be dedup-eaten by a recent hook record — correct, the hook copy already seeded `active_tasks` (#150). **The probe is ALSO ongoing liveness, not just admission (#220):** after each refresh (initial seed / 250ms rescan / 60s poll — the three sites that re-snapshot `live`) the watcher emits `AgentEvent::ProofOfLife` for every vouched id (`emit_proof_of_life`, after `scan_root` so fresh admissions already have slots); the reducer keeps a PRIVATE `recent_proof_of_life` map (deliberately not an `AgentSlot` field — no semver surface) with a 150s TTL (`PROOF_OF_LIFE_TTL`, 2.5× the 60s poll: two missed polls + slack) that `sweep_stale`'s candidate pass consults — a vouched slot is never staleness-swept while the process lives (motivating case: a permission-parked CC session renders Active after attach-replay — its hook-only Waiting is unreconstructable from JSONL — and `STALE_ACTIVE_TIMEOUT` would reap it mid-decision). **The vouch also shields the vouched slot's actively-DELEGATING subtree**: the probe never vouches subagent ids (their stems are `agent-<id>`, not session UUIDs) and a permission-parked parent renders Active, not Waiting (so `has_waiting_ancestor` can't exempt its blocked-but-live child) — `sweep_stale` therefore also skips a slot when any ANCESTOR (`scope::has_ancestor_where`, the cycle-guarded walk `has_waiting_ancestor` shares) holds a fresh vouch (`vouch_fresh`, the ONE TTL predicate) AND a non-empty `active_tasks` entry; the active-delegation gate preserves the 30-min idle backstop for completed lingering children (the b1 chained-dispatch residual). When the live signal disappears (registry entry removed / rollout fd closed) emissions stop and normal sweeps resume after the TTL lapse — subtree shield included. **And since #223 the probe drives the NEGATIVE vouch — vouched-then-gone is a high-confidence exit.** Probe failure is now EXPLICIT: `LivenessProbe` returns `Option<ProbeSnapshot>` (`ids` + `pid_of`, the id→owning-pid binding for the exit watch). `None` = the enumeration itself FAILED and the watcher changes NOTHING (admission keeps the previous snapshot, miss windows neither open nor age, no `ProofOfLife` — the reducer's TTL absorbs the gap); `Some` with empty `ids` = a healthy "nothing alive" observation (meaningful!), so absence of an id only counts inside a `Some` snapshot. A previously-vouched id MISSING from two healthy snapshots ≥ `NEGATIVE_VOUCH_MIN_SPAN` (60s — immune to Codex's brief drop-and-reopen fd gap on a write failure and to initial-seed/250ms-rescan adjacency) apart gets a Jsonl-tagged `SessionEnd` — the exit signal the CLI never writes. For Codex an fd close IS session end (including a thread switch within one process: the old rollout closes, the new one opens and gets its own vouch); CC's clean exits already `SessionEnd` via the hook, so this catches the abrupt ones (Ctrl-C, terminal close, kill) in ~60–120s instead of the 10–30 min stale-sweep. The confirmation also un-claims `seen` for the session's tracked paths (mirroring the decoded-SessionEnd revive) so a later resume/append re-registers through `emit_first_sight` — a wrongly-ended live session self-heals on its next write or re-vouch. Per-probe failure semantics: CC — unreadable-or-MISSING registry dir = `None` (ambiguous on a machine where CC runs), empty-but-readable dir = `Some(empty)`; Codex — an ABSENT sessions root = `Some(empty)` (codex never run is not a failure), `None` only when the proc-table enumeration fails (`fd_probe::pids_by_name` is now `Option<Vec<i32>>`; per-pid fd failures stay non-failures); Windows: CC = `None` (no validated pid liveness); Codex = `Some(empty)` (the `fd_probe` fallback enumerates nothing — CI-pinned by the un-gated `pids_by_name_for_nonexistent_process_is_some_empty` — so nothing is ever vouched and the negative vouch can never arm). The reducer needed NO change: a `SessionEnd` for a live slot exits it (+cascade), an unknown id is a no-op, and the `ProofOfLife` exemption is evicted with the slot in `sweep_exited`. Pinned by `negative_vouch_emits_session_end_after_sustained_disappearance` / `one_missed_snapshot_does_not_end_the_session` / `probe_failure_changes_nothing` in `tests/watcher.rs` (the `with_negative_vouch_min_span` `#[doc(hidden)]` seam shrinks the span like `with_poll_interval` does the poll). **On top of the negative vouch sits the INSTANT exit (#223 rung 2):** a watcher with a probe spawns ONE `source/exit_watch.rs` watcher (a detached `std::thread` — deliberately NOT `spawn_blocking`, a forever-loop would pin a blocking-pool slot; kqueue `EVFILT_PROC NOTE_EXIT` on macOS, `pidfd_open`+`poll` on Linux, `spawn() = None` elsewhere) and registers every pid a healthy snapshot binds (`ProbeSnapshot::pid_of`, folded into the watcher-local `pid_bindings: pid → {ids}` by `refresh_probe_snapshot` — bindings are ADDITIVE per snapshot except an observed REBIND, which migrates the id to its new pid (a `codex resume` in another process; the stale binding's death must not end the live session); otherwise the vouch ladder owns "gone"). The bound process dying emits each bound id's `SessionEnd` within milliseconds through the SAME `emit_session_exit` path the negative vouch uses (one exit path, can't fork — `seen` un-claim AND a `live` admission-set purge included (the purge stops a probe-FAILURE pass's stale snapshot from re-vouching the just-ended id into a ghost replay; for the negative-vouch caller it's a no-op), so the resume self-heal is identical) and `NegativeVouch::forget`s the id so the slower rung can't re-confirm; conversely a negative-vouch confirm `unbind_session`s its id from `pid_bindings` so a codex-style long-lived process exiting later is quiet (one process owns many rollouts — a closed rollout's session ends while the process lives). Registration races resolve at the primitive: ESRCH at registration (the macOS `EV_RECEIPT` receipt / Linux `pidfd_open`) = already dead = the exit synthesizes immediately; EPERM = dropped silently; `NOTE_EXITSTATUS` is deliberately NOT requested (status is child-only and triggers xnu's cred check — plain `NOTE_EXIT` watches same-user non-children unprivileged); ENOSYS (pre-5.3 kernel) = backend off. EVERY failure (kqueue/pipe init, kernel registration, thread death, send fail) degrades to the rungs below — the exit ladder is **instant (ms) > negative vouch (~60–120s) > ProofOfLife TTL lapse + stale sweeps** — and `run()` parks a keepalive sender clone when no ExitWatch holds one, so the select! exit arm stays forever-pending instead of resolving `None` every pass. Pinned by `instant_exit_emits_session_end_when_bound_pid_dies` / `negative_vouch_confirm_unbinds_pid_so_a_later_exit_is_quiet` in `tests/watcher.rs` + the `exit_watch` unit suite; CI asymmetry (the `fd_probe` precedent): the Linux pidfd arm is CI-pinned (ubuntu), the macOS kqueue arm local-preflight-pinned (the windows job compiles only the `None` arm). ProofOfLife refreshes ONLY the exemption: it never creates a slot (unknown id = no-op — only hook tool/permission events synthesize), never touches activity state or `last_event_at` (debounce/back-fill stay event-driven), never delays `SessionEnd`/cascade (exiting slots are skipped on insert and aren't sweep candidates anyway). Pinned by the `proof_of_life_*` + `vouch*` suite in `tests/reducer.rs` + two watcher tests in `tests/watcher.rs`: `watcher_emits_proof_of_life_for_probe_live_ids` (the initial-seed emission) and `poll_arm_refreshes_probe_snapshot_and_reemits_proof_of_life` (the 60s-poll snapshot refresh + re-emission, made testable by the `#[doc(hidden)]` `with_poll_interval` builder seam). A 250ms post-startup rescan catches files missed by `read_dir` during the initial seed (APFS metadata propagation race). **Unifying the gate inside `walk_jsonl` is the #85 fix** — it used to be split (only the old `initial_seed_walk` gated; the rescan/poll/notify read first-sight files from offset 0), so when the initial `read_dir` lost the APFS race the 250ms rescan resurrected a missed ended/stale session as a phantom `SessionStart` (a real resurrect-on-restart bug, not only a test flake).
- **Watch backend: native in prod, polling in tests.** `JsonlWatcher::run` watches via `notify`'s `RecommendedWatcher` (FSEvents/inotify/…) plus the 60s `scan_root` poll backstop. Integration tests call `force_polling_backend_for_tests(interval)` — a `#[doc(hidden)]`, set-once process hook — to swap in `notify::PollWatcher` at ~25ms instead: a real FSEvents stream costs **tens of seconds per test to set up + tear down** on macOS (it was the bulk of the watcher suite's runtime, and the real-FSEvents coupling was the seam behind the #85 flake — see the line above). A second `#[doc(hidden)]` seam, the `with_poll_interval` builder, shrinks the 60s `scan_root` poll backstop per-watcher (default unchanged in prod) — the poll arm's probe refresh + ProofOfLife re-emission are untestable at the production cadence. Both backends impl `notify::Watcher` and feed the SAME channel, so `run`'s `select!` loop is backend-agnostic. The poll uses **`with_compare_contents(true)`** (hash-based) on purpose: `PollWatcher`'s default change-date diff misses rapid appends/truncate-rewrites (the `watcher_does_not_consume_partial_trailing_line` + `watcher_resets_cursor_on_truncation_below_cursor` cases). This seam swaps only the *delivery* mechanism, never the logic — the first-sight gate stays covered watcher-free by the deterministic `walk_jsonl` unit tests. Don't wire a real FSEvents watcher into tests.
- **A hook event from an UNKNOWN session id REGISTERS it — hooks are proof of life.** A hook can only come from a live process, so a Hook-transport `ActivityStart`/`ActivityEnd`/`Waiting` whose `AgentId` has no slot means a LIVE session is invisible: its transcript was gated at first sight (mid-attach, idle >1h — the watcher edge above), and a session parked on a permission prompt appends NOTHING to its transcript, so it has no JSONL revival path at all. **Since #221 the registration normally lands with REAL identity:** `decode_hook_payload` (one payload → a `Vec<AgentEvent>`) attaches an `AgentEvent::Identity { source, session_id, cwd: Option<PathBuf> }` AHEAD of every tool/permission activity event (PreToolUse / PostToolUse / Notification / PermissionRequest, plus the reasonix custom decoder's activity arms — its cwd is the session key, so `Identity` mirrors its SessionStart exactly and coalescing holds), and the reducer's `Identity` arm registers-or-back-fills from it: no slot → `register_slot` with the real source/session_id/cwd (normal `cc·repo`-style label; a cwd-less payload — Codex PermissionRequest, CC PostToolUse — registers ordinal-labeled but with the same reap exemption as below); slot exists → the shared `backfill_identity` helper heals EMPTY source/session_id/cwd first-wins (the same heal the duplicate-SessionStart arm runs — but label upgrades stay on the SessionStart path: `Identity` carries no label authority, and it never touches activity state or `last_event_at` — the paired activity event right behind it does). The blank path is now the FALLBACK: `reducer::apply`'s pre-pass 0 (`synthesize_hook_registration`) still performs the registration for identity-LESS hook activity (`Stop` decodes without `Identity` — an end proves nothing worth registering — and directly-constructed events), starting the slot BLANK: empty source/session_id/cwd, bare ordinal label (`#N`) — deliberately NOT an unknown-cwd ghost (exempt from the 3-min `STALE_UNKNOWN_CWD_TIMEOUT` reap: the motivating permission-parked session emits no follow-up event within 3 min, so the ghost reap would kill it before any back-fill). The next real `SessionStart` (a JSONL revive, Codex's per-prompt re-emit) **back-fills** the blanks in the duplicate-SessionStart arm: empty source/session_id/cwd are healed first-wins (via `backfill_identity`), and the label is upgraded ONLY while it's still a derivation fallback (`is_fallback_label`: bare prefix / `cc#N` ordinal / the source-less `#N` hook-synthesis shape — a basename- or Rename-derived label is real information and is never clobbered). Three boundaries are load-bearing and PRESERVED by #221: (1) **JSONL events must NOT synthesize** — a transcript line can be a historical replay (the first-sight gate exists precisely for those), so the unknown-id no-op stays on that transport (`jsonl_event_for_unknown_id_stays_a_no_op`) and a JSONL-transport `Identity` is a structural no-op (`jsonl_identity_is_a_no_op` — nothing in-tree emits one; the guard IS the boundary); (2) `SessionEnd`/`Rename` don't synthesize either — an end/rename for an unknown agent proves nothing worth showing, end-ish events (`Stop`/`SessionEnd`) carry no `Identity`, and the `Identity` arm honors the SessionEnd tombstone (`hook_identity_respects_session_end_tombstone`); (3) the hook-wins dedup record is gated on the slot EXISTING post-synthesis — a registration REFUSED on desk exhaustion must not record its `tool_use_id`, or the stale record would dedup-eat the JSONL `ActivityStart` that finally registers the session once a desk frees inside `HOOK_WINS_WINDOW` (`refused_hook_registration_does_not_poison_dedup_for_the_later_jsonl_copy`; `Identity` carries no `tool_use_id`, so its refusal can't poison anything — `hook_identity_desk_refusal_is_quiet`). Pinned by the proof-of-life + G4 back-fill + `hook_identity_*` suites in `tests/reducer.rs` (`hook_activity_start_for_unknown_id_synthesizes_slot_and_goes_active`, `duplicate_session_start_backfills_hook_synthesized_slot`, `hook_identity_registers_unknown_id_with_real_identity`, …) and end-to-end by the mid-attach scenario suite in `tests/watcher.rs`.
- **Agent removal needs a `SessionEnd`; abrupt exits have none and fall back to the slow stale-sweep.** A slot is GC'd `EXIT_GRACE_WINDOW` (4.5 s) after `exiting_at` is set, which only an `AgentEvent::SessionEnd` or `sweep_stale` sets. CC emits `SessionEnd` ONE way: the **hook** (best-effort — 200 ms write timeout, no retry, races CC's own teardown, so it drops silently); a dropped hook falls back to the idle sweep. There is NO durable JSONL fallback: modern CC persists no `/exit` record at all (slash commands are `type:"system",subtype:"local_command"` lines, and no transcript ends with an /exit user line — 135-transcript corpus), and the old content-based `<command-name>/exit</command-name>` matcher false-positived on messages QUOTING the wrapper, so it was removed — user-controllable content must never drive lifecycle (`cc_session_ended` now matches only structural `session_end`/`SessionEnd` markers, which CC doesn't write today; the upstream-drift watch flags it if that changes). **Ctrl-C / terminal-close / kill produce no signal either** (the process dies before the hook runs), and **Codex has no `SessionEnd` concept at all** (its hooks are `UserPromptSubmit`/`Stop`/`PermissionRequest`; `Stop` = turn end → Idle, not session end). Those exits are reaped only by `sweep_stale` (Active 10 min / Idle 30 min / Waiting 60 min) — except where the liveness probe runs (macOS/Linux standard layouts): since #223 its NEGATIVE vouch (see the watcher edge above) ends a vouched-then-gone session in ~60–120s, leaving `sweep_stale` as the fallback for the probe's blind spots. For **CC** this is largely inherent — its clean exits usually signal via the hook, so the CC sessions that reach the long timeout are genuinely live-but-idle, abruptly killed, or the rare dropped-hook clean exit; a short CC silence-reaper would evict legitimately-idle live sessions (lunch-break idle), so don't add one. **Codex gets a carve-out** because it has NO exit signal of ANY kind: no `SessionEnd` hook (its upstream `HookEventName` enum has none — only `Stop` = turn end), no PID in any payload, and its internal `EventMsg::ShutdownComplete` is *not persisted* to the rollout (`should_persist_event_msg` → false). All three verified against upstream `openai/codex`. So `sweep_stale` is Codex's ONLY reaper for EVERY exit, and at the 30-min idle timeout an exited `cx·` sprite ghosts for half an hour. Codex Idle therefore uses the shorter `STALE_SHORT_IDLE_TIMEOUT` (5 min, in `reducer.rs`, selected by the registry-derived `SourceCaps::short_idle_reap()` = `!has_exit_signal && resurrects_on_prompt` — Codex is the only row where both hold) — safe *specifically* for Codex because the only false-positive (a live-but-idle session past 5 min) is **self-healing**: its next `UserPromptSubmit` re-emits `SessionStart` and the sprite walks back in. **Since #220 the FD probe's `ProofOfLife` exempts a vouched Codex slot from this reap** — a slot whose rollout a live `codex` process still holds open is never short-idle-reaped; the reap effectively fires only once the probe stops vouching (process exited → fd closed → emissions stop), with the 5-min timer measured from the LATER of the last event and the vouch lapse (`PROOF_OF_LIFE_TTL` = 150s after the last emission). The self-healing property is unchanged — it now covers only the probe's blind spots (non-macOS/Linux platforms, a non-standard `--codex-sessions-root`). Pinned by `codex_vouched_idle_slot_outlives_short_idle_reap`. CC keeps the 30-min `STALE_IDLE_TIMEOUT` (its real `SessionEnd` hook makes a short reaper there all cost, no benefit). Pinned by `codex_idle_agent_reaps_faster_than_claude_idle` in `tests/reducer.rs`. **A stale-swept parent now cascades its whole subagent subtree** via the shared `scope::cascade_exit` (the parent↔subagent tree ops live in `state/scope.rs`; same `parent_id` BFS the `SessionEnd` arm uses) — so an abrupt-exit parent takes its subagents with it rather than orphaning them on their own longer stale timers. **Two guards stop the cascade from evicting a *live* subagent** (modeled on k8s liveness-vs-readiness): (1) **liveness up-propagation** — a working subagent keeps its ancestors fresh (`refresh_lineage`, see the `active_tasks` edge above); (2) **readiness exemption** — a subagent blocked under a `Waiting` ancestor (its permission `Notification` was attributed to the parent, so the parent is `Waiting` while the subagent stays `Active`) is skipped by `sweep_stale` via `has_waiting_ancestor` — "not ready", not dead. So the stale-sweep cascade only fires once the whole subtree is genuinely silent. **A subagent that COMPLETES is removed promptly too** (no 30-min linger): CC writes no completion marker, so it's inferred — when the parent's last `Task` drains (`active_tasks[parent]` empties) the delegated subtree has returned, and `scope::cascade_exit(parent)` marks it exiting (the parent keeps running). **The b1 cascade is grace-DEFERRED, not immediate** (`B1_CASCADE_GRACE`, 2.5s, #151): a parallel second Task dispatch is suppressed as a leak and tracked only via its JSONL copy, so an immediate cascade at the first Task's drain would evict the second's LIVE subtree while that copy is still in watcher latency — unrecoverably (`exiting_at` has no clearer; after the GC the child's JSONL events no-op forever). Any Task insert inside the grace cancels the pending cascade (`pending_b1_cascades`); the drain's own tool_result write triggers the notify that replays the backlogged dispatch line, so one FSEvents hop is all the grace must cover. Don't "fix" the ~2.5s completed-subagent linger back to immediacy — and note the linger is unbounded for CHAINED dispatches: a sequential re-dispatch inside the grace defuses the pending entry, so the previous wave's completed children wait for the NEXT drain-to-empty (worst case the 30-min idle sweep) — the unavoidable price of never evicting a live subtree, since the cascade is whole-subtree. Residuals (consciously accepted, see #151): the chained linger above, and a sustained notify outage spanning the grace (both writes missed → 60s poll) can still evict early. A source with a clean completion signal would drive the same cascade through its own decoder; the scope ops stay source-agnostic.
- **Resurrect-in-place starts from clean correlation state.** A `SessionStart` on an EXITING root slot cancels the walkout (`fsm::resurrect_in_place`) AND evicts the dead life's `active_tasks`/`gated_before_waiting`/`pending_b1_cascades` entries (mirroring `sweep_exited`, which never ran because the corpse never GC'd — a leftover tuid can never drain, so it would suppress every hook event of the new life); the `recent_proof_of_life` vouch deliberately survives (the resurrecting process is alive by definition).
- **Codex subagents (`spawn_agent`) are wired via the `SubagentStart`/`SubagentStop` HOOKS, not JSONL paths.** Unlike CC (whose subagent has a `<parent>/subagents/agent-*.jsonl` transcript → `detect_parent_id` derives the parent from the path), a Codex subagent's rollout is FLAT (`~/.codex/sessions/YYYY/MM/DD/rollout-*.jsonl`), so the JSONL watcher renders it but ORPHANED. The hooks are the only carrier of the parent link: `SubagentStart`/`SubagentStop` payloads carry a distinct `agent_id` (the subagent, == its rollout-filename UUID) plus the parent `session_id`. `decode_hook_payload` keys the CHILD on `agent_id` (coalescing with that rollout) and emits `SessionStart{parent_id: session_id}` / `SessionEnd{agent_id}` — so the Codex subagent joins the SAME `scope` tree (cascade/liveness/readiness) as a CC one. Because the two transports race, `reducer::apply`'s `SessionStart` arm **enriches** an existing orphan's `parent_id` (rather than no-op-on-exists) when the JSONL slot was created first; it never re-parents an agent that already has a parent, and it **refuses any link that would close a `parent_id` cycle** (`scope::would_create_cycle`, checked at this ONE seam for both registration and enrichment — warn + degrade to parentless, #238): with cycles unable to exist, the scope walks need cycle guards only for termination, never correctness (a mutual-Waiting 2-cycle would otherwise exempt both members from `sweep_stale` forever). The two `CODEX_EVENTS` registrations (`install/codex.rs`) are therefore live, not dead — captured wire format is regression-pinned in `tests/sources/codex/mod.rs`.
- **Subagent display names come from `attributionAgent` in JSONL.** The decoder strips the plugin prefix (`feature-dev:code-explorer` → `code-explorer`) and emits `AgentEvent::Rename` so labels read meaningfully. Parents get their `cwd` basename instead. `cc_derive_label` falls back to the CC **project dir** (its name encodes the cwd path, `'/'→'-'`) when a Rename's seed line has no `cwd`, so an empty-cwd Rename never degrades `cc·dotfiles` back to a bare `cc`.
- **`AgentSlot.state_started_at` is `std::time::SystemTime`** — process-local in practice (no wall-clock anchoring), but the type is already serializable, so the v2 daemon split won't need a type swap. The pose system computes elapsed time relative to it for animation timing.
- **`ActivityState::Active` ≠ "tool is currently executing".** CC fires PreToolUse → PostToolUse around every individual tool call, so without debouncing the slot flickers Active/Idle on every tool. The reducer treats `ActivityEnd` as "arm pending idle" (`AgentSlot.pending_idle_at`) instead of an immediate flip; the actual transition to `Idle` is realized by `reducer::tick` after `ACTIVE_GRACE_WINDOW = 1500 ms`. Any `ActivityStart` inside the window cancels the pending idle. Net: the slot reads as continuously Active for chained tool work, and visible Idle lags real Idle by up to 1.5 s + the 1 s sweep interval (≈ 2.5 s worst case). Don't add code that depends on `Active → Idle` being instant.
- **The reducer's permission `Waiting` resolves on the gated tool's PostToolUse.** A `PermissionRequest`/decision sets `Waiting`; the matching tool's `ActivityEnd` (PostToolUse) clears it (`gated_before_waiting` tracking). It is also cleared by `SessionEnd` + retained-eviction; `gated_before_waiting` is evicted at BOTH sites — `tick`'s `retain` and `sweep_exited`'s explicit `remove` (the apply-path eviction, pinned by `gated_before_waiting_evicted_on_apply_path_sweep`). A THIRD clear site lives on the Task-drain path (#152), CONDITIONAL on the drained tuid matching the gate: the drain deliberately skips the main arm (a Waiting parent keeps Waiting), so its gate entry would go stale and a late out-of-window JSONL replay of the same END would false-resolve a still-pending permission. The conditionality is load-bearing — an unconditional clear would wipe a parallel ordinary tool's gate (pinned by `task_drain_keeps_parallel_ordinary_tool_gate`). Keep the sites in sync if you touch eviction.
- **A meeting room narrower than `MEETING_FURNITURE_MIN_W` (compute.rs) has NO sofa/table/seats — bare floor, BY DESIGN.** Below it the 16px sofa body leaves too little margin for the coarse 4×4 router to reach the seats, so an idle agent sent to sit would TELEPORT (find_path None → straight-line fallback). The room Bounds/walls/door still exist; only the unroutable furniture is dropped (same degradation the dense floor uses when too short). So "meeting room exists" no longer implies "meeting slots exist" — don't "fix" the missing sofa at small sizes. Guarded by `pathfind::tests::every_wander_waypoint_is_routable_on_the_coarse_grid` (coarse-grid reachability across seeds × sizes — stronger than the pixel-BFS connectivity sweep).
- **Occlusion is EMERGENT — there is no `occludes_behind` field / synthetic cap any more (deleted).** An overhanging piece (`visual.h > footprint.h`: plant canopy, booth column, TV monitor, whiteboard panel, pantry counter) has its footprint **south-anchored to the sprite base** by `mask::stamp_overhang_aware`, so a walker parks DEEP behind it and the piece's own y-sorted sprite paints over their lower body. The old `paint_furniture_back_floor` floor-snapshot cap is gone. **Two intentionally decoupled footprints:** the MASK uses the shallow `footprint` (occlusion + collision); `stand_point`/`approach_point` derive the USER's park distance from the full `visual` (`half_extents`/`approach_footprint`) so someone *using* the furniture parks clear of the whole sprite, not inside it. This is the industry Y-sort pattern (base-collision + sprite-occludes); **don't re-add a cap** — make the sprite overhang its shallow footprint instead. Pinned by `pixtuoid::tui::pixel_painter::tests::every_pod_occludes_via_overhang` (in the binary renderer, not core).
- **Pantry counter blocks only a shallow `PANTRY_FOOTPRINT_DEPTH` south strip, not its full sprite height.** The counter is a ¾-view sprite centered on `pos`; only its south base contacts the floor (the receding cabinet tops overhang, invariant #6). `mask.rs` south-anchors a 3px strip; a character behind it is occluded by the counter's own y-sorted sprite, couch-style — the same emergent occlusion as every overhanging piece (above). `stand_point` parks the USER clear of the full `visual`, not this shallow strip (the runtime-sized pantry was the original mask-vs-approach decouple, now generalised to all overhanging obstacles).

## Where to look

- "Where does a furniture's footprint / visual size / approach side / dwell come from?" → **`core::layout::decor`** is the single source of truth, and there is now exactly ONE geometry table. `FurnitureDef { footprint, visual, occupies_pos, dwell, approach: ApproachSides }` + `furniture_def(Furniture)` — keyed by the unified `Furniture` geometry enum — plus `desk_furniture_def()` (the home desk: same `FurnitureDef` type, NOT a `Furniture`/waypoint row, since it's owned per-agent). **The role enums carry NO dimensions**: `WaypointKind` (wander destination), `PodDecor` (aisle filler), `PlantKind`, `WallDecor` each expose `.furniture()` mapping onto one `Furniture` row, so overlapping items (a phone booth that is both a waypoint and pod decor, a plant that is both free-standing and pod decor, a whiteboard that is both wall and pod decor) have their shape defined exactly once. The singleton / per-room furniture (meeting sofa-body + table, pantry table/stool, lamp, side table) are ALSO rows now — `Furniture::{MeetingSofaBody, MeetingTable, PantryTable, PantryChair, FloorLamp, LoungeSideTable}` — distinct from the `MeetingSofa`/`MeetingStand` SEAT rows (3 seats sit on 1 body, so the seats carry `None` footprint and the body carries the obstacle). The ONLY geometry living outside the table is `PLANT_FOOTPRINT` (the one value shared by the four plant rows, named once to avoid duplication) and the linear wall thickness `WALL_THICK_H/V` (walls are a thickness, not a point-centred rect). **`footprint` is the BLOCKED ground area** (z=0 projection) — `mask::build_walkable_mask` stamps it (no inline literals); **`visual` is the sprite size** — render centering + the z-sort south row (`center_pin_south_offset`) derive from it. **Three load-bearing distinctions:** (1) `footprint` (blocked ground) ≠ `visual` (sprite, may overhang — `Furniture::PlantTall` footprint 6×**3** (a shallow south-anchored pot strip) vs visual 6×10; overhanging pieces have a footprint *shorter* than the sprite so the canopy/column occludes a walker behind it via z-sort; vertical wall footprint 1px vs 3px drawn); (2) `occupies_pos` = the agent sits/stands ON the cell (couch/meeting) vs approaches it from a side (appliances); (3) **placement overlap** (furniture buried in a wall = bug, see `freestanding_decor_does_not_overlap_room_walls`) ≠ **occlusion overlap** (glass-over-sofa, sofa-over-sitter, canopy-over-walker = physically correct, resolved by the tui z-sort, NOT a placement error). `ApproachSides` (canonical facing-South, rotated by `allows`) is the editable entry-side knob — flip one bool to drop a side. The unifier is "one geometry table (footprint + visual) + one shared z-sort," not a second `size()` per role enum.
- "How is the office laid out?" → `core::layout::SceneLayout::compute_with_seed` is the coordinator; it calls 4 private helpers: `compute_pod_desks` (pod grid), `compute_pod_decor` (aisle decor), `compute_room_walls` (wall segments + door gaps), `compute_waypoints` (all waypoint assembly). Re-exported as `tui::layout::Layout`. `core::pose::derive` for pure state→pose mapping including the Idle wander state machine (`WANDER_CYCLE_BASE_MS=7000` + per-agent jitter).
- "How does the per-agent state machine work?" → **two stacked layers, both extracted into named modules so the reducer reads as a coordinator.** **Layer A** = `state/fsm.rs`: the single-slot transitions (`enter_active`/`enter_delegating`/`enter_waiting`/`arm_pending_idle`/`settle_to_idle`/`mark_exiting`/`rename`), each independently unit-tested; `accumulate_active_ms` lives here once (was copy-pasted 4×). **Layer B** = `state/scope.rs`: the parent↔subagent tree ops. **Invariant:** `fsm.rs` mutates ONE slot; the **cross-slot correlation** (`active_tasks` suppression, `gated_before_waiting` permission, the hook-wins dedup) stays in `reducer.rs` — it spans slots and is NOT the FSM. The reducer decides *whether/which* transition fires; `fsm` decides *how* the slot changes. Don't move the correlation maps onto `AgentSlot`.
- "Why is the subagent's sprite the right one and not the parent?" → `reducer::Reducer::apply` does subagent-leak suppression via `active_tasks` before applying. `claude_code::decode_cc_line` emits `AgentEvent::Rename` from `attributionAgent`.
- "How does multi-source decoding work?" → `JsonlWatcher` takes `LineDecoder` and `LabelDeriver` fn pointers (Strategy pattern via fn pointers, not traits). CC wires `claude_code::decode_cc_line` + `cc_derive_label` (plus the optional `with_liveness_probe` seam → `live_cc_session_ids` — see the first-sight-gate sharp edge); Codex wires `codex::decode_codex_line` + its label fn (plus the same probe seam → `live_codex_rollout_ids`, open-FD binding via `fd_probe`); Antigravity wires `antigravity::decode_ag_line` + `derive_ag_label`. **Reasonix is HOOK-ONLY** (`line_decoder: None` in its registry row) — no `Source` impl, no watcher: its v2 session files are full-rewritten per turn (untailable) and its hook payloads carry no session id/path to coalesce on, so `reasonix::decode_rx_hook_custom` (cwd-keyed AgentId) runs as the row's `hook.custom` decoder, BEFORE the shared CC/Codex field requirements (its envelope is camelCase with an `event` discriminator and no `session_id` — an alien-envelope custom decoder claims EVERY event). `decoder.rs` holds shared utilities (`describe_tool_target`, `make_tool_detail`, `decode_hook_payload`). Each source owns its own format knowledge. `AgentId::from_parts(source, key)` namespaces IDs per source. Labels show source prefix: `cc·project` / `cx·project` / `ag·project` / `rx·project`. **CLI attribution comes ONLY from the shim-owned `_pixtuoid_source` key, NEVER the public `source` field** (CC overloads `source` for the SessionStart *reason* — startup/resume/clear — and reading it splits the agent into an un-reapable ghost; see the WHY at `source/decoder.rs` `decode_hook_payload`). Per-source CROSS-CUTTING facts (label prefix, hook id-key/custom decoder, reducer capability flags, the harness's line decoder) live in ONE `SourceDescriptor` row in `source/registry.rs` — the reducer's `source_label_prefix`/`stale_threshold` and the fixture harness read the table instead of name-matching, and `SourceCaps::short_idle_reap()` derives the Codex short-idle policy from wire facts (`!has_exit_signal && resurrects_on_prompt` — Antigravity lacks the signal too but can't resurrect, hence its long window). Adding a CLI = (1) the source module (SOURCE_NAME + decoders), (2) ONE descriptor row in `registry.rs`, (3) the name in `source::REGISTERED_SOURCES` (a bridge test pins table↔list equality; the fixture conformance test forces a fixture under `tests/sources/fixtures/<name>/`), (4) wire it into `pixtuoid/runtime/driver.rs::run_async` (the registry gates conformance tests, NOT runtime spawning — **a source registered in `registry.rs` + `REGISTERED_SOURCES` but NOT wired into `driver.rs` passes every conformance/manifest test yet never spawns at runtime; `driver.rs` is the ONE step the registry can't force, because each source struct has different typed override fields**) — a hook-only CLI skips (4) and ships an `install/` target instead so its hooks reach the shared socket. (5) a row in `site/src/sources.json` — single-sources the README "Supported Tools" glimpse + the site's tool × OS matrix (run `just gen-readme`); its `supported` set is pinned to `REGISTERED_SOURCES` by `tests/supported_sources_manifest.rs`, so a newly registered source FAILS that test until its manifest row exists. (6) if the CLI has a custom home/config root, add a `pub fn <cli>_home()` in `source::<cli>` honoring the CLI's own `*_HOME` env precedence (like `codex_home` → `CODEX_HOME`-if-exists-else-`~/.<cli>`), and call it from BOTH the watcher's `default_paths()` AND the installer's `default_config_path()` — one function, two consumers, so the watched root and the installed-hook config can't disagree. You do NOT separately edit `source_label_prefix`, `stale_threshold`, or the harness's `decoder_for` — those read the table. Ship a fixture exercising the **SessionStart hook** so `tests/sources/conformance.rs`'s one-AgentId assertion guards against the ghost. (Test-layout detail + add-a-CLI test steps live in the nested [`tests/CLAUDE.md`](tests/CLAUDE.md).)
- "Why don't old idle sessions show on startup?" → `source::jsonl::should_seed_at_eof` (the first-sight gate inside `walk_jsonl`). Checks `check_session_ended` (tail-scans last 8 KB for `session_end`/`SessionEnd` markers) and seeds the cursor at EOF for any file with mtime > `DEFAULT_INITIAL_WINDOW` (1 hour) — no `SessionStart`. An mtime outside the 1-hour window **OR** a `session_end` marker is the freshness gate, and it runs on every first-sight path (seed / 250ms rescan / 60s poll / notify), not only startup. Exception: a session the **liveness probe** vouches for (CC's `sessions/<pid>.json` registry; Codex's open-rollout FD binding — see the sharp edge above) is never gated, however old its mtime.
- "How does walk-pace physics work?" → `pixtuoid_core::physics` (pure, no terminal deps) — `WalkIntent` tags the walk kind (Entry/Exit/WanderOut/WanderBack/SnapBack); `walk_profile(path_len_octile, intent, agent_id)` returns a frozen `WalkProfile` with trapezoidal/triangular kinematics (triangular when path < `v²/a`, trapezoidal otherwise); `walk_progress(p, elapsed_ms)` emits `t_x1000 ∈ [0,1000]`; `walk_arrived` gates the Walking→Seated/AtWaypoint flip including the per-agent arrival pause. Constants: `V_CRUISE_COMMUTE = 0.36` octile/ms (Entry/Exit), `V_CRUISE_SNAPBACK = 0.65` octile/ms (SnapBack — the urgent Idle→Active return rushes back), `V_CRUISE_WANDER = 0.25` octile/ms (amble, wander legs), `WALK_ACCEL = 6.5e-4` octile/ms² (~0.55 s ramp; `L_crit ≈ 199` octile commute, `≈ 96` octile wander), `WALK_ACCEL_SNAPBACK = 2.0e-3` octile/ms² (~3× — short snap-backs are accel-limited, so the rush comes from a harder accelerator, not cruise). Per-agent personality: `speed_mult` (bits 24..34 of splitmix64 hash, range 0.85–1.20) and `pause_ms_for` (bits 40..52, range 200–400 ms) — disjoint from each other and from `cycle_ms_for`/`personality_for`. Near desks arrive and sit before far desks (natural staggered arrivals). The *stateful* timeline that consumes these profiles lives in `pixtuoid/src/tui/motion` (the render authority) — see that nested guide.

## Keeping the decode mapping current (upstream drift)

Tool names, hook-event names and transcript types are upstream wire contracts that change **without notice** (the `Task`→`Agent` rename shipped undocumented). Four defenses, in order of reliability:

1. **Resilient detection (in code).** Prefer a *semantic* signal over a hardcoded name where one exists — e.g. `make_tool_detail` keys subagent dispatch on the stable `subagent_type` input field, not the tool name. This survives a rename by construction.
2. **Self-monitoring from the real stream (in code).** The decoders already see ground truth. An unrecognized `hook_event_name` `bail!`s + `warn!`s; a dispatch under an unknown name logs a `debug!` breadcrumb. These surface drift the moment it appears in a user's own usage — no network, no upstream-layout dependency.
3. **Internal consistency test.** `every_registered_codex_event_decodes` / `every_registered_cc_event_decodes` (in `install/{codex,claude}.rs`) assert that every hook event we *register* has a decoder arm — catching the *registered-but-not-decoded* class (the original SubagentStop bug) at `cargo test` time.
4. **Upstream watch (CI).** `scripts/check_upstream_drift.py` + `.github/workflows/upstream-drift.yml` (weekly) read the names we depend on directly from our source and compare to live upstream; on a vanished name or a new unhandled Codex/Reasonix hook it opens a deduped `upstream-drift` issue. Sources:
   - **Codex** (open Rust): `codex-rs/protocol/src/protocol.rs` `HookEventName` enum (raw on `main`). `PreCompact`/`PostCompact` are intentionally omitted (`CODEX_KNOWN_OMITTED` in the script) — not agent activity.
   - **CC** (closed binary — no source to grep): the docs markdown `code.claude.com/docs/en/tools-reference.md` (authoritative tool-name list). Tool names like `Agent` are NOT schematized upstream, so defense #2 is the real backstop for CC.
   - **Reasonix** (open Go): `internal/hook/hook.go` `Event` consts + the payload json tags (`event`/`cwd`/`toolName`/`toolArgs`/`message`) on `main-v2`. `PostLLMCall`/`PreCompact`/`SubagentStop` are intentionally omitted (`REASONIX_KNOWN_OMITTED`).
